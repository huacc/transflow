# PDF 年报页面画像与千问分类树：语料研究 v0.1

> 状态：研究草案，尚未替换《PDF 翻译排版独立引擎总体设计》的正式分类树。
> 目标：让千问按可见页面特征逐级分类，并让每个叶子稳定映射到排版工具箱。

## 1. 研究范围

本轮只研究页面分类和工具箱边界，不修改翻译引擎、分类 Prompt 或现有总体设计。

全局图片约束：

- 图片像素及图片内部的文字、数字、图例和说明不翻译、不重排、不覆盖；
- 图片的位置、尺寸、颜色和内容不修改；
- 只有独立存在的 PDF 原生文字对象才进入翻译与排版修复；
- 千问可以读取图片内容用于判断页面角色和版式，但不得把图片内容转成翻译或修复任务。

语料：

```text
D:\项目\开源项目\MerqFin\spikes\独立测试\样本\年报
```

全量扫描结果：

- 200 份 PDF；
- 41,185 页；
- PDF 全部可打开，页面读取错误为 0；
- 36,794 个粗粒度页面几何签名；
- 8,039 页位于重复几何签名中，说明正文、财务附注等模板页大量重复；
- 原生文字启发式结果约为：英文 10,778 页、中文 9,245 页、中英混排 20,979 页、原生文字近空 183 页。该统计只用于抽样，不是人工真值。

在全量特征扫描后，按封面、目录、末页、原生文字为空、正文密度、栏型、财务表、复杂混排、财务摘要、经营指标、概览和结构图等 16 个分层组，视觉复核 110 张真实页面截图。

## 2. 研究原则

### 2.1 分类服务于修复，不服务于文档主题

页面属于医药、地产、银行或 ESG 报告，不决定排版工具箱。分类只关心：

1. 页面承担什么角色；
2. 可翻译文字由哪种排版结构承载；
3. 翻译后文字允许在哪些容器内流动；
4. 哪些对象不可动；
5. 哪种工具负责修复，怎样验收。

### 2.2 只有工具行为不同才增加叶子

如果两个类型使用相同工具、相同约束和相同质量裁决，不应拆成两个叶子。

如果修复方式不同，即使代码有所重复，也应拆开，避免修改一个通用工具影响其他页面。

### 2.3 路由节点与观测维度分离

会改变工具箱的维度进入分类树；只改变工具参数的维度作为观测值。

例如：

- `text/table/chart/diagram` 会改变工具箱，是路由节点；
- 精确栏数、密度、语种通常只改变参数，是观测值；
- 栅格文字会改变执行能力，但不改变页面角色，属于能力门禁。

## 3. 从语料中确认的关键事实

### 3.1 页序只能辅助判断

- 第一页通常是主封面，但不能单独依靠 `page_number=1`。
- 最后一页可能是结束页，也可能仍是财务表、五年财务摘要或财务附注正文。
- `00863_OSL GROUP_英文_2025.pdf` 第 193 页是五年财务摘要表，不是结束页。

因此页序只能作为辅助证据，不能直接产生角色标签。

### 3.2 原生文字为空不等于页面没有可见文字

原生文字近空的抽样页同时出现了：

- 图片内含标题的章节页；
- 图片内含口号的整页照片；
- 真正无可翻译内容的装饰空白页；
- 整页正文已栅格化、肉眼可见大量文字的页面。

例如：

- `00050_HK FERRY (HOLD)_英文_2025.pdf` 第 218 页是真正装饰空白页；
- `02803_br_09803_PP BEDROCK_br_PP BEDROCK-U_英文_2025.pdf` 第 25 页是栅格化正文，不是无文字页。

因此分类阶段必须比较“截图中是否存在可见文字”和“PDF 是否存在原生文字对象”。视觉判断由千问多模态完成，不使用独立 OCR 作为分类前提。

但执行阶段只处理 PDF 原生文字对象：图片内可见文字即使能够被千问识别，也不翻译、不重排。整页正文已经栅格化时，仍可被分类为 `body`，但该页执行结果是原样透传。

### 3.3 图片对象数量不能直接代表图片主导

不少年报将背景、底纹或整页底图保存为大图片对象。即使页面存在覆盖整页的图片对象，正文仍可能是可编辑文字。

因此图片数量和图片面积只能作为证据，必须和截图、文字对象共同判断。

### 3.4 bbox 栏聚类不能直接决定正文栏型

抽样发现：

- 双语左右对照正文可能被 PDF 合并为少量大文字块，导致规则漏判多栏；
- 三列表格会产生多个稳定 x 区间，导致规则误判为多栏正文；
- 姓名加履历、标题加边注也可能产生伪第二栏。

栏型必须在排除表格单元格、标题、边注和图片标签后，判断是否存在稳定正文阅读流。

### 3.5 密度不能先于页面结构分类

“稀疏文字”抽样中同时出现了：

- 普通正文；
- 财务表；
- 分区目录；
- 章节视觉页；
- 大面积留白的指标表。

因此密度不是正文类型。只有到达 `flow_text` 或 `anchored_blocks` 后，密度才用于选择字体、字号、行距和压缩策略。

### 3.6 混合页面是年报常态，不是异常

语料中反复出现：

- 正文 + 局部表格；
- 表格 + 大段附注；
- 财务表 + 小型柱状图；
- 公司简介 + 组织/股权结构图；
- 双语正文 + 表格。

若两种结构都含有重要可翻译内容并需要不同修复工具，不能强制塞入单一类型。

## 4. 建议的分类结构 v0.1

### 4.1 执行门禁：可编辑文字范围

该节点不决定页面角色，只决定后续工具是否具有编辑能力。

```text
page.editable_text_scope
├─ native       可翻译内容主要由 PDF 原生文字对象承载
├─ hybrid       页面同时有原生文字和图片内文字，只处理原生文字
├─ image_only   页面内容全部或基本全部位于图片中，整页原样透传
└─ none         页面没有需要翻译的原生文字，整页原样透传
```

判断证据：匿名页面截图、原生文字块、原生文字覆盖区域。截图用于区分页面角色和 `image_only/none`，不能把图片文字加入翻译范围。

执行影响：

- `native`：进入常规文字/表格/图表工具箱；
- `hybrid`：只处理原生文字对象，图片及图片内部内容保持原样；
- `image_only`：不翻译、不重排，原页直接透传；
- `none`：不进入翻译排版修复，原页直接透传。

### 4.2 一级角色树

```text
page.role
├─ cover        主封面或章节封面
├─ contents     主目录或分区目录
├─ body         承载实际正文、表格、图表、结构图或信息块
├─ end          品牌落款、联系信息或结束视觉页
└─ visual_only  无可翻译文字的照片、装饰或空白页
```

#### cover

- 以报告/章节身份标题和主视觉为主；
- 实质正文很少；
- 主封面与章节封面可记录 `cover.scope=document|section`，但先共用工具箱。

正例：`01093_CSPC PHARMA_英文_2025.pdf` 第 1 页。

#### contents

- 多组重复“条目 + 目标页码”；
- 主要功能是导航；
- 可以出现在文档中部，不能只看页序。

正例：`00235_中策資本控股_中文_2025.pdf` 第 3 页、`02215_德信服務集團_中英合刊_2025.pdf` 第 2 页。

强反例：`01745_LVJI TECH_英文_2025.pdf` 第 7 页是主席报告正文，虽然正文中出现 `contents` 单词，但不存在目录条目与页码关系。

#### end

- 页面功能是结束文档，通常只有品牌、公司名称、联系信息、认证标记或结束视觉；
- 最后一页只作为辅助证据；
- 存在财务表或实质正文时不得判 `end`。

正例：`01967_信懇智能_中文_2025.pdf` 第 246 页、`00482_聖馬丁國際_中英合刊_2026.pdf` 第 206 页、`08092_ITE HOLDINGS_中英合刊_2025.pdf` 第 144 页。

#### visual_only

- 页面没有报告身份、导航关系或实质正文功能；
- 常见于纯照片、装饰过渡页或真正空白页；
- 图片内存在报告标题时仍可能是 `cover`，存在章节标题时仍可能是章节封面，存在栅格正文时仍应判 `body`；图片内容是否可编辑不改变页面角色。

正例：`00050_HK FERRY (HOLD)_英文_2025.pdf` 第 218 页。

### 4.3 正文排版所有权树

仅在 `page.role=body` 后进入：

```text
body.layout_owner
├─ flow_text
│  └─ body.flow_text.columns
│     ├─ single
│     └─ multi
├─ table
├─ chart
├─ diagram
├─ anchored_blocks
└─ composite
   ├─ text_table
   ├─ text_diagram
   └─ table_chart
```

`composite` 当前只保留语料中已经观察到、且需要不同工具协同的组合。以后只有出现重复样本并证明工具行为不同，才增加新组合。

确定性归约器另保留 `body.freeform` 兜底目录：当页面已确认属于 `body`，但 `body.layout_owner` 的规则、千问初判和一次细粒度复核仍无法稳定归类时进入。`freeform` 不作为千问的普通候选，不等同于 `anchored_blocks`，也不直接承诺正式工具箱；它用于收集分类树缺口并驱动下一轮调整。

#### flow_text

页面主体由一条或多条稳定文字阅读流承担。段落、列表、双语左右栏和多栏条目都可属于 `flow_text`。

工具行为：

- 文字只能在原阅读流容器内回流；
- 多栏不得跨栏串写；
- 标题、页眉页脚、边注不参与栏数判断。

正例：

- 单栏：`01528_RS MACALLINE_英文_2025.pdf` 第 3 页；
- 多栏：`01380_中國金石_中英合刊_2025.pdf` 第 127 页、`02119_捷榮國際控股_中英合刊_2025.pdf` 第 49 页。

强反例：`01717_AUSNUTRIA_英文_2025.pdf` 第 73 页含规则财务网格，不能因横向存在多个文本区间判为多栏正文。

#### table

主体信息绑定于行、列、表头和单元格；大部分可翻译文字属于单元格。

工具行为：

- 行数、列关系、单元格绑定和左上锚点不动；
- 按单元格翻译与适配；
- 不允许文本跨单元格流动。

正例：`01717_AUSNUTRIA_英文_2025.pdf` 第 73 页、`03988_中國銀行_中文_2025.pdf` 第 11 页。

强反例：`00585_意力國際_中英合刊_2025.pdf` 第 152 页标题包含“财务报表附注”，但页面主体是双语连续正文，不是表格。

#### chart

主体是用柱、线、扇区、坐标轴、图例或数据标签表达数值关系的图表。

工具行为：

- plot、柱线扇区、坐标关系和颜色不动；
- 只处理独立 PDF 文字对象形式的标题、图例、轴标签和数据标签；
- 烙在图表图片内部的文字、数字、图例和标签全部保持原样；
- 标签必须保持原数据绑定。

正例可继续使用独立测试样本中的中英文柱状图页。年报语料中的 `01528_紅星美凱龍_中文_2025.pdf` 第 6 页属于“表格 + 小型经营图”，更适合作为 `table_chart` 边界例，而不是纯 `chart`。

#### diagram

主体是组织结构、股权关系、流程、地图标注或时间关系图；存在节点、连线、空间对应或地图位置，但不一定存在数据坐标轴。

工具行为：

- 图形、节点位置、连线和指向关系不动；
- 只处理独立 PDF 文字对象形式的节点标签、说明和图注；
- 图片内部的节点文字和说明保持原样；
- 不把节点文字合并成正文流。

正例：`00987_中國再生能源投資_中文_2026.pdf` 第 2 页的项目地图。

#### anchored_blocks

替代当前含糊的 `freeform`。页面主体由多个独立 KPI、指标、卡片或说明块组成，没有单一稳定正文流，也没有表格网格、图表坐标或结构图连线所有权。

工具行为：

- 每个信息块独立翻译和适配；
- 文字框左上锚点固定；
- 禁止跨块流动；
- 背景、颜色和图片不动。

正例可使用当前穿刺样本中的 KPI 面板和公司概览页。三列日期事件页具有稳定纵向阅读流，应归 `flow_text.multi`，不是 `anchored_blocks`。

#### composite

当两种结构都承担大量可翻译内容、都需要独立工具修复，且任一结构都不是可忽略附件时进入 composite。

当前已观察组合：

| 叶子 | 语料例子 | 工具顺序 |
|---|---|---|
| `text_table` | `01244_思路迪醫藥股份_中英合刊_2026.pdf` 第 227 页 | 固定正文区域和表格区域；分别调用 flow_text 与 table 工具 |
| `text_diagram` | `00305_五菱汽車_中英合刊_2026.pdf` 第 2 页 | 先固定结构图，再处理正文流和节点标签 |
| `table_chart` | `01528_紅星美凱龍_中文_2025.pdf` 第 6 页 | 表格单元格与图表标签分别修复，禁止相互侵占 |

实现不需要通用区域图。V1 只需保存有限、固定的区域清单：区域 bbox、区域类型、工具箱和执行顺序。

## 5. 叶子与工具箱映射

| 分类叶子 | 工具箱 | 核心不变量 |
|---|---|---|
| `cover` | `cover` | 主视觉、报告/章节身份和标题锚点 |
| `contents` | `contents` | 条目与页码绑定、引导关系和对齐 |
| `end` | `end` | 品牌、联系信息和结束视觉 |
| `visual_only` | 无修复工具 | 原页直接保留 |
| `body.flow_text.single` | `body.flow_text.single` | 单一阅读流内适配 |
| `body.flow_text.multi` | `body.flow_text.multi` | 原栏内适配，禁止跨栏 |
| `body.table` | `body.table` | 单元格、行列、网格不变 |
| `body.chart` | `body.chart` | 图形和数据绑定不变 |
| `body.diagram` | `body.diagram` | 节点、连线和空间关系不变 |
| `body.anchored_blocks` | `body.anchored_blocks` | 独立文字框锚点和块间隔离 |
| `body.composite.*` | 对应有限组合工具箱 | 固定区域、固定顺序、跨区域零影响 |
| `body.freeform` | 暂无正式工具箱 | 二级无法稳定归类的可追踪兜底，待人工复核分类树 |

## 6. 不进入叶子的观测维度

以下字段先作为观测，不形成类别笛卡尔积：

```text
source_language       zh | en | mixed | none | unknown
language_layout       single_language | parallel_bilingual | mixed_inline
column_count          1 | 2 | 3 | ...
density_band          low | medium | high
cover_scope           document | section
editable_text_scope   native | hybrid | image_only | none
embedded_objects      table/chart/diagram/image 的辅助存在情况
```

只有后续实验证明某个观测值需要完全不同的工具和质量裁决时，才升级为分类节点。

## 7. 千问逐级裁决协议

### 7.1 一次只判断一个节点

推荐执行顺序：

```text
检测原生可编辑文字范围；截图内容只用于分类
→ page.role
→ 若 body：body.layout_owner
→ 若 flow_text：body.flow_text.columns
→ 若 composite：body.composite.kind
→ 到达叶子后再判断语种、精确栏数和密度
```

### 7.2 每个节点的输入

千问只接收：

- 匿名页面截图；
- 当前节点名称和允许子项；
- 当前节点所需的少量原始证据；
- 已确认父路径；
- 当前节点的正例、强反例和边界例。

图片证据只用于分类。传给翻译器、PagePatch 和修复工具的文字清单必须来自 PDF 原生文字对象，不得包含千问从图片中读出的文字。

禁止输入文件名、路径标签、人工 expected label、工具实现和下一层候选。

### 7.3 正例与反例必须成对

每个类型卡至少包含：

- 3 个跨报告正例；
- 3 个视觉相似但标签不同的强反例；
- 2 个应返回 `INCONCLUSIVE` 的边界例；
- 中文、英文和中英混排尽量均有覆盖；
- 同一公司中英文版本必须进入同一数据划分，防止模板泄漏。

### 7.4 双路裁决

每个节点保留：

1. 可审计规则只根据结构证据输出判断；
2. 千问只根据截图、原始证据和 exemplar 输出判断；
3. 两边一致则进入下一节点；
4. 不一致时只对当前节点做一次细粒度复核；
5. 仍不一致则 `INCONCLUSIVE/UNSUPPORTED`，不得默认塞入 `body` 或 `anchored_blocks`。

## 8. 下一阶段验证计划

1. 从本轮 110 张截图扩充为人工确认的节点级 gold 集；
2. 首先验证 `page.role`，再验证 `body.layout_owner`，不一次测试整棵树；
3. 对每个节点输出混淆矩阵，尤其关注：
   - contents vs 普通短条目正文；
   - end vs 最后一页正文；
   - visual_only vs 栅格文字页；
   - flow_text.multi vs table；
   - table vs text_table；
   - chart vs diagram；
   - anchored_blocks vs 多栏正文；
4. 分类稳定后，为每个叶子选择一个典型页和一个困难页做工具箱穿刺；
5. 只有工具箱穿刺证明需要新分支时，才扩展分类树。

## 9. 当前结论

现有方向“逐页、逐节点、分类后进入专用工具箱”成立，但现有 `body.content=text/table/chart/freeform` 过于粗糙：

- `freeform` 名称和边界不具备工程可维护性，应替换为 `anchored_blocks`；
- 页序、语种和密度不能混入页面角色；栅格文字可以辅助识别角色，但不进入翻译范围；
- diagram 与 chart 的修复约束不同；
