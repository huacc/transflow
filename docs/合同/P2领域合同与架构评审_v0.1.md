# P2 领域合同与架构边界评审 v0.1

评审结论：PASS

开放问题：0

## 范围

本评审仅冻结 P2 的纯领域合同、状态不变量、五个外部端口、版本化资源和架构红线；不实现 P3 的 Standalone Adapter、Checkpoint/Artifact 持久化或测试 AI，不实现 P4 的 PDF 纵向运行。

## 冻结结论

1. 生产入口只接受一份完整 PDF 的 `DocumentRunRequest`；列表、目录形态和非 PDF 路径由领域合同拒绝。
2. `TranslationBundle` 必须与请求的 `unit_id` 集合和顺序完全一致；缺失、重复、新增、改写和重排均失败。
3. `PagePatch` 在应用前同时校验 `source_hash`、`page_no`、`geometry_hash` 和 `owner`。
4. Job、Page、Checkpoint 和 Repair 预算均由显式状态函数控制；所有页面进入 `FINALIZED` 前不得最终化文档。
5. 对外依赖只允许 `JobQueuePort`、`CheckpointPort`、`ArtifactPort`、`TranslationPort`、`ModelDecisionPort` 五个端口；语义翻译与其他模型判定分离。
6. 设计中的 17 条分类路由在 Taxonomy 和 Catalog 中各出现一次。当前没有生产晋升清单，因此全部路由保持禁用；spike 阶段证据仅作为来源证据，不等同于生产启用证据。
7. 领域层只能依赖标准库和领域层自身；Ports 只能依赖标准库、领域层和 Ports 自身。生产包禁止引用 spike、宿主项目、模型代理框架、浏览器 HTML 回填和单页 PDF 拼接方案。

## Adapter 映射冻结

- Standalone 与宿主任务 Adapter 共同实现 `JobQueuePort`。
- `read_control()` 统一返回 `ControlSignal`。
- 不增加 Freeform 或 ProductProjection 专用 Port；相关能力通过既有领域合同和五个端口组合表达。

## 取舍说明

- P2 使用标准库 `dataclass` 和 `Protocol`，避免把数据库、HTTP 客户端、PDF 引擎或第三方 Schema 实现带入纯合同层。
- 两份 JSON Schema 作为跨进程载荷边界；Python dataclass 负责进程内不变量。JSON Schema 无法表达的 `unit_id` 顺序一一对应由 `TranslationBundle` 领域校验强制执行。
- Catalog 的启用条件同时要求 `evidence_status=PASS` 和 `promotion_manifest_present=true`，防止把实验通过误判为生产晋升。
