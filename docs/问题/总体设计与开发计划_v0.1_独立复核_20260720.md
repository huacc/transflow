## 0. 总判断

1. **9C/9A/9B 的插入逻辑成立。** 硬顺序 `P9 → P9C → P9A → P9B → P10` 中每条依赖都是真实消费关系：P9A 消费 G9C 的 Route/owner 与完整性合同，P9B 的 PageRepairMemory identity 绑定 G9A 的 document memory hash，P10 的已知失败叶（visual_anchored、composite）消费 G9B 的多轮修复闭环。无循环依赖；"翻译→布局→翻译"的死锁环被 `target_layout_policy` 从源基线+语言对+受控字体+配置派生（不等待译后页面）显式切断。Gate 全部是可数指标（调用数 `0`、构建数 `1`、越权 `0`），可客观验证。
2. **harness + loop engineering 的引擎设计方向正确，与穿刺已验证组合一致**（见 §3）。显式状态机 + Port/Adapter + 有界 Decision Node、不引入通用 Agent 框架，是当前证据下的正确取舍，不是过度设计。
3. **主要问题分三类：**
   - 设计文档把少数**穿刺中并不存在的能力**表述成了已有资产，其中"唯一 Patch 解释器 + 源副本串行回放"是最终化主链，实际需要一个未单列的逆向迁移工程（§2 问题 P-1）。
   - 若干**迁移等价基线的事实缺口**，不补会让"等价"证据失真（§2 问题 P-2/P-3/P-4/P-5）。
   - 对"打通优先"的 V1，**跨文档/跨运行学习层**（RepairRuleRegistry 晋级、PriorRepairEvidenceRef、PromotionEligibility）以及 **Freeform 区域恢复** 属于可降级项（§4）。这部分与同目录已有评审结论一致，本文补充 Freeform 一项和 G14 代表性一项。

## 1. 9C/9A/9B 逻辑合理性的逐项核对

### 1.1 P9C（纠偏）：动机真实，但两个事实前提需要修正

- 纠偏动机真实。P9 阶段报告（`docs/reports/P9阶段_*_20260720_155043.md`）显示六叶均为 `PASS_DISABLED_WITH_FALLBACK`、12 份真实千问候选 1 接受 11 回退——"HTTP 200 ≠ 翻译成功""出了 PDF ≠ 产品验收"的问题真实存在，SemanticUnitMap + 完整性门禁 + 双轨 + 三类结论是正确药方。
- **问题 P-6（小）**：P9C.1 任务描述称旧经验文档存在"硬失败不产候选"与"完整译文必须产诊断候选"的合同冲突，并预期其进入 contradiction register（P9C.1-T04）。实际核对 `spikes/page_toolbox_engine_puncture_v1/docs/经验/` 十份文档：不存在"硬失败不产候选"的说法，相反 P13/P14/P15/P16 经验已明确要求失败也要物化诊断候选、禁止源页复制冒充、Provider 未产译文记 `NO_TRANSLATED_CANDIDATE`。即新合同与穿刺经验是**继承关系**而非冲突关系。冲突若存在，应登记其真实出处（可能是 P5–P9 阶段报告的口径），否则 P9C.1-T04 会在错误前提上找东西。
- **问题 P-7（小）**：P9C 的"前向合同不回写历史 Gate"纪律自洽，但 P8/P9 已迁移叶子要改造为消费 SemanticUnitMap/完整性门禁（设计 §9 对 LegacyToolboxAdapter 的要求），这部分适配量包含在 P9C/P10 回归里，没有单独估算。逻辑没问题，排期时别把 P9C 当成纯合同文档阶段。

### 1.2 P9A（文档级布局记忆）：骨架是打通必需，内容可先薄后厚

- **必需的部分**：生产输入是整本 PDF，必须枚举页；页眉/页脚/页码的跨页重复证据必须全书扫描；多页并发必须有一个只读共享快照防串扰。这部分是"文档级引擎"的定义性需求，穿刺 100% 单页，无法提供，是真实的头号绿地。
- **可延后的部分**：`target_layout_policy` 的角色画像统计（分布/置信度/区间）是一致性优化，不是正确性需求。V1 可以只放字体回退链 + 受控调整范围。与同目录评审 O-1 建议一致：先保"全页屏障 + 单次构建 + 只读共享 + 页级允许不同有效布局"骨架，字段先薄后厚。
- **延迟代价已被设计认知**：全页事实/路由屏障把分类延迟放到整本关键路径上，§25 风险表已登记并以流式聚合+确定性 fallback 闭合屏障缓解。自洽，但应在 G13 容量证据里单独呈现"屏障前耗时"。

### 1.3 P9B（页级修复记忆与多轮重排）：内核是迁移，外壳是新建

- 内核（多轮 accept/rollback、预算、无改善停止、action/state 去重）在穿刺有原型（`shared_pdf_kernel/repair.py` 的 RepairController、`multi/tools/orchestrator/typography_repair_loop.py` 的 repair memory），迁移合理。
- **问题 P-4（接缝，本评审新增）**：统一的 `RepairCoordinator` 要做"排除已尝试动作、每轮只选一个动作"，就必须能**枚举每个叶子的修复原子目录**并生成确定性 action_key。但 P7.1 的 `LegacyToolboxAdapter` 把 repair 包成叶内黑盒（六阶段外形里的 `repair()`），协调器看不到动作空间。计划没有任何一个子任务负责"叶私有修复原子 → 协调器可枚举的 RepairProposal 目录"这个映射合同：P9B.1 定义的是 Rule IR/Registry（跨文档规则），不是叶内原子目录。这是 P9B 实施时第一个会炸的接缝，建议在 P9B.1 增加一个明确的"叶修复原子登记合同"子任务（哪怕是手写清单 + 版本 hash），否则 RepairCoordinator 只能退化为"叶自己循环、协调器记账"，G9B-3/G9B-4 的"重复动作拦截、单动作归因"证据将无法产生。
- **问题 P-8（小）**：`max_no_improvement=2` 依赖质量向量的数值比较，但"无改善"的比较容差（数值抖动多少算无改善）全文未定义。机械 Judge 是确定性的，问题不尖锐，但 Finding 计数型向量在边界样本上仍可能抖动，建议在 P9B.1 冻结比较规则（相等或容差）。

## 2. 设计/计划 vs 穿刺现实：事实核对发现的偏差

按严重程度排序。这些偏差不推翻设计，但会让"迁移等价"证据和工作量估算失真。

### P-1（最重要）"唯一 Patch 解释器 + 源副本串行回放"不是穿刺已有能力，是新建主链

- **设计表述**：§2.2 把 `PagePatch` 和 `shared_pdf_kernel` 的"写入"列为已验证资产；§9 要求"候选页和最终文档使用同一 Patch 解释器"；§12.3 规定最终 PDF 从源 PDF 副本串行回放已批准 PagePatch。
- **穿刺现实**（已逐文件核实）：
  - `shared_pdf_kernel/patch.py` 的 `apply_page_patch()` **只被 `scripts/run_p2_kernel_validation.py` 引用**；toolboxes/ 下没有任何叶子使用它。
  - 每个叶子有自己的 `renderer.py`（如 `chart/tools/renderer.py` 458 行），直接调 fitz `add_redact_annot`/`insert_textbox`，消费自己的 `LayoutPlan`。**穿刺模型是"候选渲染即最终写入"，不存在"候选只用于裁决、最终从源副本回放声明式 Patch"的分离**。
  - 穿刺 `PagePatch`（contracts.py:248）**没有 owner 字段**（owner 只在 `Finding.owner`），设计 §9 的"Patch 绑定 owner/region"是扩展。
- **影响**：把 N 个叶子的自定义 renderer 逆向为一个声明式、可序列化、带 owner/geometry 绑定的 Patch 操作集，是整个迁移中**最隐性、工程量最大**的一段，且没有独立阶段为其负责：P4.0/P6 建解释器，P8–P10 逐叶迁移，"renderer → Patch 的逐叶逆向映射"夹在两段之间没有明确 owner。迁移等价测试（P8.2-T01 等比较 Patch/Finding/verdict/渲染）只能在叶子已经被 Patch 化之后运行，不能证明逆向过程本身。
- **建议**：在 P6 或 P7 增加明确的逐叶"renderer → Patch interpreter 逆向映射"工作项与等价证据（渲染像素/结构 diff 在冻结容差内），并在迁移台账中把"统一 Patch 解释器"登记为**新建**而非 Lift。这不改变设计方向（回放模型本身是对的，避免页级 PDF 拼接的坑），只是别再把它当已有资产估算工作量。

### P-2 分类控制流存在高置信规则短路，迁移等价基线需覆盖

- 设计 §20.5 描述为"rule → qwen primary → conflict 时一次 review → deterministic resolver"。
- 穿刺实际多出短路（`engine.py:137-164`、`resolver.py:33-43`）：layout_owner 节点规则凭 TABLE1/BTABLE1 直接证据且置信度 ≥0.9 时**不调千问**；规则高置信时 **HIGH_CONFIDENCE_RULE 直接采用、不复核**。
- P5.3-T01（"规则明确；预期不调用模型"）部分覆盖，但匿名基线的逐叶指标必须包含短路分支的口径，否则迁移前后指标不可比。低严重度，补上即可。

### P-3 分类 REVIEW 注入"带标签正例图"，去身份化策略有一个未决矛盾

- 穿刺复核阶段会把 exemplars 中**其他样本的已确认标签 + 样本图**（如 `{"label":"anchored_blocks","sample_id":"P0003"}`）放进模型载荷（`engine.py:118-126,182-199`）。
- 设计 §20.3 要求从"模型输入、Prompt 和 exemplar 中移除文件名、sample_id、gold 标签及其可推导信息"。但正例图的**标签文本**是复核的参照物：删标签则复核失去功能，留标签则人工标签继续进入模型输入，与去身份化目标冲突。设计/计划都没有显式决策这一点。
- **建议**：在 P5.1/T2 显式决策（可选项：保留"标签+匿名图"作为合法 few-shot 并登记为豁免；或改为无标签图 + 标签定义文本）。这是低严重度但必须留痕的决策点。

### P-4 toolbox spike 依赖声明不完整，P1 锁定时勿照抄

- 穿刺 `pyproject.toml` 只声明 `httpx`，实际 import 还包括 `fitz`(PyMuPDF)、`numpy`、`PIL`(Pillow)、`pdfminer`。P1 冻结锁文件时以实际 import 为准，不要从 spike 的 pyproject 继承依赖清单。

### P-5 MerqFin 基线漂移已经触发

- 设计核验基线为 `main@3d831c5`（2026-06-14）；本地 checkout 已是 `ecf30dc`（2026-07-05）。设计自带的机制（M0/P15 重新冻结）已覆盖，此处仅确认该机制现在就已处于"必须执行"状态，不是理论风险。`claim_next_job` 不按格式过滤、`translation_pages`/`translation_job_pages` 双表现状经抽查属实。

## 3. harness + loop engineering 评审：方向正确，细节如上

对照当前 agentic 系统的最佳实践，逐项评价：

**做对的部分（不要动）：**

- **确定性 harness 与有界模型节点分离**。LLM 只出现在 ClassificationDecision 一个节点（Rule IR 的 `llm_question` 是 V2 候选），其余全部是确定性代码。任务领取、租约、CAS、Checkpoint、进程隔离都在 harness 层，不交给模型。与两个穿刺已验证的组合一致。
- **loop 的完成定义前置**：SemanticUnitMap 在翻译前冻结分母，TranslationCompletenessDecision 是进入布局的唯一门禁。"loop 什么时候算完成"有客观判据，这是大多数 agent 系统最缺的一件东西。
- **每轮从不可变源页重建完整候选**，不在失败候选上累积涂改——消除 drift 的正确做法。
- **每轮只选一个主要动作**——因果可归因，accept/rollback 比较才有意义。
- **接受条件 = 目标 Finding 改善 且 无硬回归**——双向比较，防止"修了溢出、踩了保护对象"。
- **action/state 去重 + 连续无改善停止 + 预算硬上限 + 分层 fallback（区域→页→整本→源副本）**——保证任何输入都有终止态，这是"失败也必须出结果"的机械保证。
- **版式 Repair 复用同一 TranslationBundle**——成本与确定性的双重控制，只有翻译合同失败才定向重译。
- **诊断/交付双轨**——可观测性产物不污染发布链，源副本不能冒充翻译候选。
- **不引入 LangGraph/CrewAI/AutoGen 的论证**（§21.2/§21.4）写了明确的重新评估触发条件，是负责任的"不采用"决策，不是懒惰。

**需要补强的点（即 §1/§2 的 P-4、P-8，此处不重复）。**

## 4. 过度设计评估（相对"打通优先"的 V1）

与同目录已有评审一致的部分（简述，论证见其原文）：RepairRuleRegistry + Rule IR + 晋级流程（G9B-8）是跨文档学习系统，穿刺零基础，V1 只留可为空的只读快照占位；PriorRepairEvidenceRef 九指纹精确匹配在生产几乎不触发（G9B-10），降为审计只读；PromotionEligibility 轴 V1 无消费者，常量占位；P9C.4 大规模字形/数据绑定回归移出合同阶段；TargetLayoutPolicy 保留冻结但内容从薄。

**本评审新增两项：**

### P-9 Freeform 区域恢复（P11）是 V1 最大的可选新建，建议降级为待决策

- 穿刺对 freeform 的处理是"整页透传"（UnclassifiedPageRecovery 已对 `page.role` 失败这么做）。设计 §11 的区域分解 + 私有 Adapter + 合并器是**全新建设施**（穿刺零代码、零自然样本），工作量与 P9A 同级。
- 从"打通"角度，freeform → 整页透传 + 诚实记录已满足"每页必有终态、失败也出结果"；区域恢复提升的是翻译覆盖率，是质量目标而非打通目标。当前分类审计的 freeform 率是 0/457（虽然是整理后样本，不代表生产分布）。
- **建议**：V1 把 P11 降级为"freeform → 整页透传 + 证据记录"，区域分解标记 `[待决策]` 由负责人按生产 freeform 率预期裁决；或裁剪为只支持 single/multi 两类正文区域的最小分解。无论哪种，都不影响 9A/9B/9C 的正确性内核。

### P-10 G14 的"真实翻译产品验收"对第二阶段代表性有限，需补一道生产路径复验

- G14-14 要求"至少一份完整真实 PDF 经 migration-only Qwen Adapter 达到 ProductAcceptance PASS"。但该 Adapter 不在生产 wiring，其 prompt 与第二阶段 `transflow.page_units.v1` 不同。也就是说 G14 的产品验收证据**不能代表**第二阶段真实 AI 路径的产品质量。
- 计划已有 P19.6 覆盖分类侧复验，但翻译侧只有 P19.2 的 E2E 打通，没有阈值化的 ProductAcceptance 复验。
- **建议**：在 P15.5 集成阈值注册表中补一项"生产 AI 路径 ProductAcceptance 复验"（同口径：语义/物化覆盖 100%、漏项 0、Quality PASS），纳入 G19 或 G20 量化清单。成本很低，堵的是"G14 PASS 后产品质量结论被偷渡到生产"的口子。

### P-11（提示，非缺陷）V1 交付时应显式管理"启用叶很少"的预期

- P9 结束后只有 visual_only、single（及可能的 chart/diagram）可达启用标准，多数叶 disabled → 真实年报大量页面透传 → TranslationCoverage 偏低。这是诚实且正确的 V1 取舍（G14-14 只要求一份文档 PASS），但 G14 报告应显式写明"产品验收范围 = 当前已启用叶"，避免"工程闭环 PASS"被读成"年报能翻了"。设计的三类结论机制已具备表达能力，用之即可。
