> `docs/计划/Transflow_PDF翻译排版引擎_详细开发计划_v0.1.md`（P9C 2088–2233、P9A 2235–2375、P9B 2378–2535）

## 0. 一句话结论

9C/9A/9B 的**插入点、硬顺序和依赖关系是正确的，没有循环依赖，Gate 可客观验证，纠偏补齐的动机成立**。
主要问题不在"逻辑对不对"，而在"对一个以打通为目标的 V1 是否过重"：**过度设计集中在跨文档 / 跨运行的"学习与复用"层**（RepairRuleRegistry、规则晋级、PriorRepairEvidenceRef），而**文档内的正确性内核（SemanticUnitMap + 完整性门禁、诊断/交付双轨、单文档只读布局记忆、页级有限修复循环）是恰当的、且大部分能复用穿刺已有资产**。

建议：保留正确性内核，把学习/复用层降级为 V2 或"合同占位、实现延后"。

## 1. 已核实的事实前提（这些都成立，不是问题）

评审先验证了三份文档赖以成立的事实，避免在错误前提上讨论逻辑：

1. **穿刺成熟度声明是诚实的。** 逐个核对 toolbox spike 的 `stage_gate.json`：所有叶子均为 `NON_BLIND*` 状态、`promotion_manifest_present: false`；单叶 `body.flow_text.single`=PASS(EXPERIMENTAL)、`chart`/`diagram`=PASS_NON_BLIND、`multi`/`table`=NOT_EVALUATED、`visual_anchored`/`anchored_blocks_chart`=FAIL、其余多为 EVIDENCE_INSUFFICIENT。与设计 §2.3 完全一致。设计没有夸大穿刺资产。

2. **状态机是穿刺已有状态机的忠实超集。** 穿刺 `state_machine.py` 的 `PageState` 与设计 §13.2 `PagePipelineState` 对齐；设计新增的 `TRANSLATION_READY`（完整性门禁分支）和 `FALLBACK_READY` 是增量，不是重构。

3. **翻译完整性检测在穿刺里已有成熟实现，9C 是"提升"不是"发明"。**
   - `multi/tools/translation_validation.py:19` 的 `canonicalize_with_targeted_retry` 已实现"一次只重译一个有直接错误证据的容器"。
   - `anchored_blocks/tools/translation_guard.py` 的 `RequiredLiteralRetryProvider` 已覆盖 `source_language_residue`/`incomplete_translations`/`invented_placeholders`/缺失字面量。
   - 结论：P9C.2 的完整性裁决约 **60–70% 的检测实质可复用**，真正新增的是"统一的 SemanticUnitMap 分母抽象 + 进入布局前的统一门禁位置"。

4. **页级修复循环 / 修复记忆原型在穿刺里已有。**
   - `shared_pdf_kernel/repair.py:6` 的 `RepairController` 已是通用多轮 accept/rollback（max_rounds、max_no_improvement）——可直接作为 RepairCoordinator 内核。
   - `multi/tools/orchestrator/typography_repair_loop.py:46` 的 `new_typography_repair_memory` 已跟踪 `attempted_action_keys`、`seen_state_hashes`、`attempts`、`terminal_reason`，并做 STATE_CYCLE / 耗尽检测、落盘 JSON。这就是 `PageRepairMemory` 的原型。

一句话：**9C/9A/9B 想补的洞是真实存在的，且内核大量站在已验证资产上。**

## 2. 逻辑合理性：插入点、顺序、依赖（评价：成立）

- **硬顺序 `P9 → P9C → P9A → P9B → P10` 合理。** 计划第 2097 行明确 "C" = Corrective（纠偏），不是排在 P9B 之后——这是一个防误解的显式说明，值得肯定。
- **依赖是真实的，非人为串联：**
  - P9A.1 消费 G9C 的 Route/owner 合同（计划 2254）；
  - P9B.1 消费 G9A 不可变文档记忆并绑定 document-memory hash（计划 2396）；
  - P10 消费 G9B 修复闭环（计划 2557）。
- **不存在循环依赖。** 最容易出现的"要先翻译才能定布局、要先定布局才能翻译"这个环被显式切断：`target_layout_policy` 从**语言对 / 受控字体 / 配置**派生，**不等待译后页面**（设计 §9.5.1 步骤 3、计划 2244/2250/2278）。文档级记忆（不可变）→ 页级记忆（追加）的分层也避免了写回环（G9B-9：页级决策不回写文档记忆）。
- **Gate 接线干净。** 每个 G9C/G9A/G9B 验收行都引用了具体的 `P9x.y-Tn` 测试 ID，符合计划 §2.5 的"双线锁定"纪律。
- **Gate 可客观验证。** 阈值几乎都是 100% / 0 的可数指标（翻译调用数 `0`、构建次数 `1`、mock 数 `0`、越权 Patch `0`），FAIL 条件是二值可检的。

这一层没有需要修的逻辑缺陷。

## 3. 核心问题：对"打通优先"的 V1 存在过度设计

按用户"不要过度优化、本版本主要目标是打通"的约束，以下项目相对于 V1 目标偏重。**过度设计集中在跨文档 / 跨运行的学习与复用层，而非文档内正确性内核。**

### P-1（最突出）RepairRuleRegistry + 规则晋级是完整的"学习系统"，V1 不需要
- **位置：** 设计 §9.5.3 / §9.5.4；计划 P9B.1（Rule IR 合同）、P9B.4 + G9B-8（两份文档 + 一份 holdout 盲测才能晋级）。
- **问题：** 这是一套带 Rule IR（`required_facts`/`mechanical_when`/`llm_question`/`allowed_verdicts`/`salience`/冲突解析）、离线晋级流程、holdout 盲测的**跨文档经验泛化机制**。穿刺里**零基础**——所有记忆都是单页单次运行作用域。它**不给"这一页能不能修好"提供任何页级修复循环本身没有的正确性**；它优化的是"跨文档少走弯路"，属于成熟度/效率优化，不是打通所需。
- **建议：** V1 只保留"读取一份固定、可为空的 registry 快照"这个**只读接口占位**，规则数量可以为 0；把 Rule IR schema、晋级流程、holdout 盲测、G9B-8 整体推迟到 V2。这样 RepairCoordinator 仍能编译运行，且不牺牲任何页级正确性。

### P-2 PriorRepairEvidenceRef 在真实 PDF 上几乎永不触发，却背了完整 Schema+Gate
- **位置：** 设计 §9.5.5；计划 P9B.1-T08、P9B.3-T07、P9B.4-T08、G9B-10。
- **问题：** 它要求 source/page/Route/Toolbox/config/document-memory/registry/TranslationBundle/Schema/实现指纹**九项全部精确一致**才能把旧失败 action/state 当排除集。现实中只有"对完全相同的文档、相同配置、相同引擎版本重开一次 run"才会命中——这是重放同一份失败文档的边角场景。为它建整套 Schema + 三个测试用例 + 独立 Gate 行，投入产出比很低。
- **建议：** V1 删除或降为"审计只读引用（记录存在但不参与动作选择）"，去掉 G9B-10 的排除集语义。恢复靠 §13.4 的同 run Checkpoint 复用即可，不需要跨 run 继承失败经验。

### P-3 TargetLayoutPolicy 作为"冻结文档 Artifact"增加了可延后的身份/失效面
- **位置：** 设计 §9.5.1；计划 P9A.1（身份指纹）、P9A.3（冻结/并发/失效）。
- **问题：** 把目标语言布局政策做成随 `memory_hash` 冻结、带九项指纹失效规则的独立 Artifact，对确定性和并发只读复用是有价值的；但对一个"打通"版本，它也可以在**页级布局解析时按同一份语言/字体/配置即时计算**，省掉一层文档级身份管理和失效矩阵。
- **建议：** 若为了并发确定性保留冻结，可接受；但应意识到这是"确定性优化"而非打通必需，评审时不要把它当作必做项压测。属于"可保留但需知成本"，非必删。

### P-4 PromotionEligibility 结论轴在 V1 无消费者
- **位置：** 设计 §9.6.2 / §12.1 三类结论；计划贯穿 G9C/G9B。
- **问题：** EngineeringClosure（工程闭环）和 ProductAcceptance（产品验收）在 V1 都有真实用途——它们精确表达"出了 PDF ≠ 翻译成功"，这是设计最有价值的贡献之一，**必须保留**。但 PromotionEligibility（晋级资格）服务的是"叶/规则能否启用/晋级"，其消费者要到灰度/规则晋级阶段才出现，对"打通"这一版是投机字段。
- **建议：** 三类结论**保留 EngineeringClosure + ProductAcceptance 两类**即可满足 V1 的诚实报告需求；PromotionEligibility 可先记为常量 `NOT_EVALUATED` 占位，不为它单独设计投影和回归。

### P-5 P9C.4 的字形 / 数据关系深度回归超出"合同基线"
- **位置：** 计划 P9C.4-T04/T05、G9C-8（CJK 标点 / Latin 断词 / 旋转 / CID / Wingdings / 图标-图例-数字-单位绑定 / 低对比度）。
- **问题：** 这是很好的质量硬化清单，但它是**质量覆盖深度**，不是"横切合同补齐"的必要条件。P9C 的定位是"补合同"，把大规模真实字形回归塞进合同阶段会拉长打通关键路径。
- **建议：** P9C 只保留"合同能被真实页触发一次"的最小真实性证据（避免纯 mock），把 457 页规模化字形/数据绑定回归移到 P10 之后或专门的质量阶段。

## 4. 与"打通优先"无关、但确实是最大工程量的项（非过度设计，需正视）

### O-1 DocumentLayoutMemory / 全文档 + page_no 运行是最大的绿地项
- 穿刺**100% 单页**：`sample_snapshot.py:18` 直接整文件复制并断言 `snapshot_sha256 == source`（样本本身就是预拆的单页 PDF）；`extract_page_facts` 默认 `page_index=0`；渲染/裁决实际读 `document[0]`；批处理逐样本独立、无跨页共享状态。
- 因此 P9A（文档级记忆）**以及"生产入口只接受整本 PDF、由 Coordinator 枚举页"这条设计主线**，是真实的行为迁移，工作量最大。这**不是过度设计**——它是打通"文档级引擎"的定义性需求；但排期上应认识到它 + 单页解耦是穿刺到生产的头号结构缺口，别被 9B 的记忆细节掩盖了 9A 的体量。
- 建议：优先保证 P9A 的"整本枚举 + 单次构建 + 只读共享 + 页级不同有效布局"骨架跑通，其内部字段覆盖度（角色画像分布、置信度区间等）可以先薄后厚。

## 6. 建议的 V1 裁剪清单（可直接执行）

保留（正确性内核，勿动）：
- SemanticUnitMap + TranslationCompletenessDecision + 进入布局前统一门禁（P9C.2）
- FinalDeliveryArtifact / TranslatedDiagnosticCandidate 双轨 + 发布隔离（P9C.3）
- RouteCapabilityMismatchFinding（P9C.4 的合同部分）
- EngineeringClosure + ProductAcceptance 两类结论
- DocumentLayoutMemory 骨架 + 整本枚举 + 只读并发共享（P9A 全部，体量最大但必需）
- 页级 PageRepairMemory + RepairCoordinator 有限多轮（P9B.1–P9B.3 的页级部分，复用 `RepairController` 与 typography memory 原型）

降级 / 延后到 V2：
- RepairRuleRegistry + Rule IR + 规则晋级 + holdout 盲测（P-1，G9B-8）→ 只留可为空的只读快照占位
- PriorRepairEvidenceRef 排除集语义（P-2，G9B-10）→ 删除或降为审计只读
- PromotionEligibility 轴（P-4）→ 常量占位
- P9C.4 大规模字形/数据绑定回归（P-5）→ 移出合同阶段

需正视工作量（非裁剪对象）：
- DocumentLayoutMemory + 单页解耦（O-1）→ 头号绿地，优先打通骨架

## 8. 总评

- **harness + loop engineering 的引擎设计方向是对的**：显式状态机 + Port/Adapter + 最小 BoundedDecisionRunner，不引入通用 Agent 框架（设计 §21.2），与两个穿刺已验证的组合一致，"不会漏页、不会无限循环、失败仍能最终化"更易证明。这个选型值得肯定，不是过度设计。
- **9C/9A/9B 的逻辑、顺序、依赖、Gate 都成立**，纠偏动机真实，内核站在已验证资产上。
- **唯一实质批评是范围**：把跨文档/跨运行的学习与复用层（Registry 晋级、PriorRepairEvidenceRef、Promotion 轴、大规模字形回归）从 V1 关键路径移除或降为占位，让"打通"版聚焦在文档内正确性 + 文档级记忆骨架上。这样既符合用户"不要过度优化、主要目标是打通"的约束，也不牺牲任何正确性保证。
