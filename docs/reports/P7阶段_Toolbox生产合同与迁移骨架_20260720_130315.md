## 1. 执行范围与边界

本阶段严格实现 P7 的生产合同与迁移骨架，没有迁移或启用 P8 及以后任何真实 Toolbox 叶算法。设计中的 17 条 Route 均进入 v2 显式 Catalog，全部保持 `enabled=false`，并具有唯一 `PAGE_PASSTHROUGH` 出口。

千问迁移适配器未接入本阶段生产代码。总体设计 §16.2 将真实模型限定在迁移质量测试，而 P7 的目标是冻结 Translation request/bundle 拆分与架构边界；G7 架构扫描确认 Toolbox 直接 Provider/HTTP/重试/并发调用为 0。用户提供的 Base URL、API Key 和模型名均未写入代码、配置、资源、测试或报告。

## 6. 设计质疑与采用方案

1. 计划要求 P7 提供全 Route Catalog，但 P8 才开始迁移并启用真实叶。若 P7 直接启用已有 `PASS` 穿刺叶，会违反“代码迁移不等于生产启用”。采用方案：17 条 Route 全量显式登记，P7 全部 disabled，并给每条唯一文档收敛 fallback。
2. “签名叶结论”未指定外部签名基础设施。采用方案：使用规范 JSON + SHA-256 的本地可复算内容证明，明确不声称密码学身份签名；后续如引入签名服务，可在不改变三态语义的前提下扩展。
3. P7.4 的“至少两个叶的重复证据”若只按页面重复容易吞并正文。采用方案：共享语义边缘同时要求页覆盖阈值和至少两个不同 Route 证据；正文、表注、图标签显式 handback。
