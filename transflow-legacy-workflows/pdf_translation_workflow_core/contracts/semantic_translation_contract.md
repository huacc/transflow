# Semantic Translation Contract

## Purpose

`product_quality` mode requires real target-language translations before PDF generation.
The generator must not invent placeholder text and must not silently translate by fixed rules.

The default direction is English to Chinese for backward compatibility. A run may declare a different direction, for example Chinese to English, by setting:

```json
{
  "source_language": "zh",
  "target_language": "en",
  "target_text_field": "translation_en"
}
```

## File Location

For each run case input, semantic translations are read from:

```text
docs\input\semantic_translations\<case_id>.translations.json
```

Example pattern:

```text
docs\input\semantic_translations\<input_id>.translations.json
```

The file is an input artifact for `S5_TranslationPlan` and `S7_GenerateCandidate`.
It is not generated by the PDF generator.

## D2 Batch Materialization

In `product_quality`, semantic translations must be materialized through a bounded D2 batch loop. A single unbounded prompt that asks a model or Codex executor to translate every extracted unit in one response is not a complete contract.

The required S5 sequence is:

```powershell
python pdf_translation_workflow_core\tools\planners\build_translation_batch_manifest.py `
  --source-extraction <source_extraction.json> `
  --case-id <case_id> `
  --source-language <source_language> `
  --target-language <target_language> `
  --target-text-field <target_text_field> `
  --batch-dir <run_dir>\translation_batches `
  --out <run_dir>\translation_batch_manifest.json
```

For each batch listed in `translation_batch_manifest.json`, the executor fills `D2_translation.prompt.json` from:

```text
translation_batches\<batch_id>.slot_values.json
```

Before writing model-call artifacts, the executor must first persist a passing boundary report:

```text
translation_batches\<batch_id>.workspace_boundary.json
```

Then it may persist:

```text
translation_batches\<batch_id>.prompt_instance.json
translation_batches\<batch_id>.model_output.json
translation_batches\<batch_id>.validation.json
translation_batches\<batch_id>.decision_record.json
```

Batch validation command:

```powershell
python pdf_translation_workflow_core\tools\validators\validate_translation_batch.py `
  --slot-values <run_dir>\translation_batches\<batch_id>.slot_values.json `
  --model-output <run_dir>\translation_batches\<batch_id>.model_output.json `
  --out <run_dir>\translation_batches\<batch_id>.validation.json
```

After every batch validation passes:

```powershell
python pdf_translation_workflow_core\tools\generators\assemble_semantic_translations.py `
  --manifest <run_dir>\translation_batch_manifest.json `
  --out docs\input\semantic_translations\<case_id>.translations.json `
  --evidence-out <run_dir>\translation_assembly_evidence.json
```

Then run the final semantic gate with `validate_semantic_translations.py`.

If any batch is missing, invalid, or cannot be translated, the executor may retry only that batch and must record the retry. If the missing capability remains, route to `S_FAIL_CAPABILITY`; do not generate a product candidate.

## Unit Rules

Required units are selected by `source_language`:

- `source_language=en`: every extractable source line with `ascii_tokens` is a required translation unit.
- `source_language=zh`: every extractable source line with CJK characters is a required translation unit.
- Other language codes require an explicit round-local extraction rule and must be recorded as `Ax_AdaptiveChange`.

Visible lines that are already in the target language are not required semantic translation units. If such a line would be covered by target-language recomposition, the generator may preserve it as:

```json
{
  "translation_mode": "preserve_already_target_language_span",
  "target_text": "<same visible text>",
  "reason": "visible target-language span in source PDF needs redaction/redraw to avoid overlap"
}
```

This preservation path is allowed only when the line does not contain source-language characters and does contain target-language characters. It must be reported separately as `preserved_target_language_unit_count`; it must not inflate `semantic_translated_unit_count`.

For every required unit:

- The declared `source_language` must match the dominant language inferred from the current `source_extraction.json`. A semantic JSON for another source-language direction must fail even if no units are selected under its declared language.
- The validator must fail when the declared source-language rule selects zero required units, unless a separate OCR/tooling boundary has already routed the case away from product-quality generation.
- `unit_id` must match the source extraction unit id.
- `source_text` must match the source extraction text exactly after trimming.
- The target text field named by `target_text_field` must be non-empty.
- `translation_target_text` should duplicate the full target text for direction-neutral consumers.
- For `target_language=zh`, the target text must contain Chinese characters.
- For `target_language=en`, the target text must contain ASCII letters and must not contain CJK residue.
- `layout_variants` are required for constrained layout roles such as compact labels, short labels, table cells, table headers, legends, chart labels, and side navigation. Fields include `compact_label_zh`, `short_label_zh`, `table_cell_zh`, `legend_zh`, `vertical_nav_zh`, `compact_zh`, `display_zh`, `compact_label_en`, `short_label_en`, `table_cell_en`, `legend_en`, `vertical_nav_en`, `compact_en`, and `display_en`.
- Numbers, percentages, years, footnote markers, and currency amounts must be preserved.
- Numeric metric labels may be normalized by generic target-language unit rules when the provider returns source-language metric words unchanged. The allowed trigger is current-unit text shape only: numbers plus percentage/per cent/percent, basis points, percentage points, pps/bps, currency units, million/billion/trillion, months, or years. This normalization must not use sample names, page numbers, fixed coordinates, reference PDFs, or document-specific glossary entries.
- Placeholder strings are forbidden: `中文回填`, `中文标题`, `中文标签`, `待翻译`, `占位`, `placeholder`, `TBD`.
- Line-category or metadata descriptions are forbidden even when they contain the target language. Examples: `本行说明...`, `本行列示...`, `This line reports...`, `This line describes...`, `当前页的财务报告、治理或业务信息`, `保留数值与标记...`, and `preserve figures/markers...`.
- Copying source text unchanged into a semantic translation target field is forbidden. The only exception is the explicit `preserve_already_target_language_span` mode above, which is preservation/redraw evidence rather than translation evidence.

`layout_variants` rules:

- Variants are D2/D4 model or human-reviewed decisions, not generator inventions.
- Each variant must match `target_language`; for Chinese it must contain CJK, and for English it must not contain CJK residue.
- Variant fields may include `compact_label_zh`, `short_label_zh`, `table_cell_zh`, `legend_zh`, `vertical_nav_zh`, `compact_zh`, `display_zh`, `compact_label_en`, `short_label_en`, `table_cell_en`, `legend_en`, `vertical_nav_en`, `compact_en`, and `display_en`.
- A variant may abbreviate wording for fit, but it must preserve the semantic identity recorded in the full target text.
- A variant must not replace real translation with a line-category description. The same forbidden metadata patterns apply to variants.
- The generator may choose a variant only when `layout_policy.json` permits that variant field for the current region kind.
- If a constrained role lacks a matching variant, the generator must not invent one. It may try the full target text through the normal fit curve. If the current layout policy explicitly permits `constrained_text_image_fit`, S7 may render the full target text as a wrapped local text image; otherwise any overflow/fallback must remain visible as `text_fit` evidence and feed D2/D4 repair.

## Provider Rules

Forbidden providers:

```text
deterministic_placeholder
placeholder
manual_placeholder
empty/null provider
```

Allowed provider names must identify the real translation path, for example:

```text
codex_gpt5_manual_semantic_translation
openai_model_semantic_translation
human_reviewed_semantic_translation
```

The provider name alone is not enough. The run must also persist prompt artifacts or human-review evidence.

## Prompt Evidence

Every model-backed translation decision must persist:

| Artifact | Required content |
|---|---|
| `workspace_boundary.json` | resolved planned prompt/model/decision/validation output paths and `workspace_boundary_verdict=PASS` |
| `prompt_instance.json` | system prompt, filled user prompt, model/provider identifier |
| `slot_values.json` | exact unit ids, source text, page/region metadata sent to the model |
| `model_output.json` | raw model output before normalization |
| `decision_record.json` | normalized verdict, coverage, confidence, next state |

These artifacts must be referenced by `prompt_artifacts` and `decision_log.jsonl`.

## Tool Enforcement

Before `semantic_backfill`, run:

```powershell
python pdf_translation_workflow_core\tools\validators\validate_semantic_translations.py `
  --source-extraction <source_extraction.json> `
  --translations docs\input\semantic_translations\<case_id>.translations.json `
  --out <run_dir>\semantic_translation_validation.json
```

`translation_validation_verdict: FAIL` blocks candidate generation and routes to `S_FAIL_CAPABILITY`.

## Generator Enforcement

`generate_semantic_backfill.py` consumes semantic translations and writes:

```text
candidate.pdf
candidate_generation_evidence.json
translations.json
layout_plan.json
layout_policy.json reference
```

It must set:

```json
{
  "translation_provider": "not deterministic_placeholder",
  "source_language": "en|zh|...",
  "target_language": "zh|en|...",
  "target_text_field": "translation_zh|translation_en|translation_target_text",
  "translation_quality": "semantic_translation",
  "semantic_coverage": "full_semantic_translation",
  "strategy": "redact_extractable_<source_language>_lines_and_insert_semantic_<target_language>_regions",
  "layout_policy_json": "...",
  "layout_policy_sha256": "..."
}
```

If semantic translations are missing or invalid, it must fail instead of substituting placeholder text.

The generator also requires:

```text
layout_policy.json
```

This policy is created in `S6_LayoutPlan` by `tools\planners\build_layout_policy.py` with the matching generic language profile and may be revised by the D4 layout prompt. The generator must not hide D4 decisions as hardcoded role thresholds, language-direction behavior, font curves, or compact-label substitutions.

## Anti-Overfit Boundary

The translation contract may use the current run's extracted `unit_id`, `source_text`, page index, role, bbox, and font size as evidence.
It must not branch on known sample filename, fixed page number, known page identity, exact coordinates, or memorized sample translations.

Official bilingual versions of the same source may be used only as offline evaluation references after the run. They must not be supplied to D2 prompt slots, semantic translation JSON generation, terminology forcing, layout policy generation, or visual adjudication during product execution. If a reference is consulted after the run, the audit report must label it as `offline_reference`, not `runtime_input`.
