# PDF 分类专用工具箱穿刺建设计划 v1.1

## 0. 一页结论

### 0.1 本穿刺解决什么问题

页面分类穿刺已经能够把单页 PDF 路由到相对稳定的分类叶子。下一阶段不再继续扩大全局万能框架，而是按分类叶子逐个建设、测试和冻结专用工具箱。

目标链路为：

```text
分类结果（只读）
  → 公共合同与机械内核
  → Codex 作为实验引擎探索某个分类的最佳工具流程
  → 千问按页翻译
  → 分类专用工程化回填、排版、裁决和修复
  → 固定译文回归 + 千问端到端回归 + 留出验证
  → 工具箱晋级为稳定 PageToolboxModule
  → 注册进完整 PDF 翻译排版引擎
```

### 0.2 核心实施顺序

```text
P0 计划与边界冻结
  → P1 穿刺运行基建
  → P2 公共 PDF 机械内核
  → P3 body.flow_text.single 首条纵向闭环
  → P4 body.flow_text.single 稳定与晋级
  → P5 body.flow_text.multi
  → P6 body.table
  → P7 body.composite.flow_text_table
  → P8 contents
  → P9 cover
  → P10 end
  → P11 anchored_blocks / visual_anchored
  → P12 chart / diagram
  → P13 其余 composite
  → P14 工具箱目录与多页文档集成
  → P15 全量回归与正式引擎移交
```

任何阶段没有通过自己的 Gate，不得把“已有候选 PDF”“脚本可运行”或“过程完整”当成阶段完成。

### 0.3 当前工程验收状态（2026-07-13）

本计划从本次验收起区分“工程验收”和“形式晋级”：工程验收表示当前实现、已生成产品和已声明能力边界获得用户接受，可以交接或继续后续穿刺；形式晋级仍要求补齐原 Gate 中尚未执行的证据、`stage_gate.json`、workflow freeze 和 `PromotionManifest`。不得把未执行项补写为已执行。

| 阶段 | 工具箱 | 工程验收 | 已知边界 | 形式晋级 |
|---|---|---|---|---|
| P4 | `body.flow_text.single` | **通过，用户验收** | 全量 278 页中 255 页机械产品 PASS；22 页能力/流程失败和 1 页产品失败保留为显式例外 | 待补 |
| P5 | `body.flow_text.multi` | **未通过，继续收口** | 最新批次产品通过面仍不足 | 未开始 |
| P6 | `body.table` | **通过，用户验收** | 当前能力聚焦具有稳定重复数值列直接证据的财务表格；修复后未按本次明确范围重跑全量 20 页 | 待补 |

P4、P6 的工程验收不追溯伪造缺失证据，也不自动解除 composite 对 P5 的依赖。后续若要注册到正式 Catalog，再单独完成形式晋级。

### 0.4 本轮最终成功标准

- 分类穿刺保持只读、独立和可复现；
- 公共层只保留跨分类稳定的机械原语和窄合同；
- 每个支持的分类都有独立工具箱、流程文档、测试样本和质量证据；
- 每个工具箱能明确返回通过、能力不足或质量失败；
- 千问按页翻译，翻译模块保持简单、独立、可替换；
- 稳定工具箱通过统一接口被文档级引擎调度；
- 多页 PDF 可拆页、逐页处理、原序合并，并保持页框和不可动对象；
- 不支持的分类显式失败，不由万能工具或自由 Prompt 猜测处理。

### 0.5 明确不做

- 不修改 `page_classification_engine_puncture_v1`；
- 不重建页面分类树；
- 不建设通用排版规划器；
- 不建设开放式全局 ToolRegistry；
- 不建设术语库、翻译记忆、多模型路由或复杂翻译平台；
- 不翻译、不重排图片内文字；
- 不为文件名、样本 ID、页码或某个绝对坐标写特例；
- 不一次性生成全部空工具箱目录；
- 不在证据不足时宣称某个工具箱稳定。

## 1. 实施前锁定决策

### 1.1 上游与工作目录

本穿刺根目录：

```text
spikes/page_toolbox_engine_puncture_v1/
```

只读上游：

```text
spikes/page_classification_engine_puncture_v1/
```

主要输入：

```text
spikes/page_classification_engine_puncture_v1/分类结果/
```

本穿刺不得修改上游分类代码、Prompt、定义、Artifact 和分类结果。选中的页面必须复制为本穿刺的版本化样本快照，并记录来源与哈希。

### 1.2 页面分类树

正式工具箱路径：

```text
cover
contents
end
body.flow_text.single
body.flow_text.multi
body.flow_text.visual_anchored
body.table
body.chart
body.diagram
body.anchored_blocks
body.composite.flow_text_table
body.composite.anchored_blocks_chart
body.composite.chart_table
body.composite.flow_text_chart
body.composite.flow_text_diagram
```

特殊路径：

- `visual_only`：原页透传，只做不可变校验；
- `body/freeform`：能力不足兜底，不默认绑定工具箱。

### 1.3 翻译边界

翻译以单页为单位：

```text
PageTemplate
  → PageTranslationRequest（本页全部可翻译原生文字，带容器 ID）
  → 千问一次处理本页
  → PageTranslationBundle（ID 对齐的页级译文）
  → 工程代码按容器 ID 回填
```

约束：

- 不合并不同页面为一次翻译请求；
- 不建设跨页翻译记忆；
- 千问只返回译文，不决定 bbox、字体、字号、行距和排版；
- 首版只检查返回数量、ID 对齐、空译文和必要数字保护项；
- 后续同事的翻译实现只要遵守同一接口即可替换当前千问适配器。

### 1.4 Codex 的阶段性角色

实验阶段由 Codex 作为实验引擎：观察失败、调用当前分类允许的工具、补工具和调整固定流程。工具箱晋级后，主流程必须固化为确定性状态机；正式运行不再由 Codex 在全局工具集中自由选择下一步。

### 1.5 Gate 纪律

每个阶段必须保存：

- 阶段输入；
- 实际交付物；
- 自动测试结果；
- 人工视觉核查清单；
- 未解决问题；
- `stage_gate.json`；
- 进入下一阶段的明确结论。

Gate 只有三种结论：

```text
通过              可以进入下一阶段
证据不足          当前能力保留，不得称稳定；可转去补样本或处理其他叶子
失败              必须修复本阶段问题，不得进入依赖它的阶段
```

## 2. 目标系统组成

### 2.1 公共层与私有层

```text
DocumentCoordinator
├─ PageDecisionImporter             读取分类结果
├─ TranslationProvider              独立的页级千问适配器
├─ PageToolboxCatalog               leaf_key → 已晋级工具箱版本
├─ SharedPdfKernel                  机械 PDF 原语
├─ QualityController                通用硬门禁 + 分类专用裁决
└─ DocumentAssembler                按原序合并页面

PageToolboxModule                   每个分类独立
├─ PageTemplateBuilder
├─ TranslationRequestBuilder
├─ LayoutPlanner
├─ ToolboxJudge
└─ RepairPolicy
```

### 2.2 可以共享的内容

- 工作区边界和源文件只读检查；
- PDF 页框、对象、文字、字体和图片事实提取；
- 源页、候选页和区域截图渲染；
- 文件和对象哈希；
- 字形覆盖与字体嵌入检查；
- PagePatch 的机械应用；
- 通用溢出、越界、遮挡和不可动对象校验；
- Artifact、状态轨迹和运行报告写入；
- 修复接受、回滚和预算控制。

### 2.3 必须由分类工具箱私有的内容

- 页面画像、正例和反例；
- 文字容器识别和翻译单元绑定规则；
- 排版参数选择和调整顺序；
- 分类特有 Finding；
- 分类特有修复策略；
- 分类特有晋级样本和质量标准。

允许不同工具箱出现重复实现。只有经过至少两个工具箱验证、且不包含页面语义的机械能力，才允许下沉到公共层。

### 2.4 核心数据合同

P1 必须冻结以下最小合同，后续阶段只能做向后兼容扩展：

| 合同 | 责任 |
|---|---|
| `SampleManifest` | 样本来源、哈希、原文档、页码、用途和标签 |
| `PageFacts` | 可复核的 PDF 直接事实 |
| `PageTemplate` | 当前工具箱对可编辑文字容器的理解 |
| `PageTranslationRequest` | 一页一次的翻译输入，包含容器 ID 和有序原文 |
| `PageTranslationBundle` | 与请求 ID 对齐的页级译文 |
| `PagePatch` | 允许修改的内容和排版动作 |
| `Finding` | 问题类型、位置、严重度、证据和 owner |
| `PageQualityDecision` | 通过、可修复、能力不足或质量失败 |
| `RunManifest` | 工具、Prompt、模型、版本、状态和 Artifact |
| `PromotionManifest` | 工具箱晋级版本及完整证据引用 |

### 2.5 中文执行状态

| 中文状态 | 内部值 | 下一步 |
|---|---|---|
| 样本已准备 | `SAMPLE_READY` | 提取事实 |
| 页面事实已就绪 | `FACTS_READY` | 构建模板 |
| 页面模板已就绪 | `TEMPLATE_READY` | 构建页级翻译请求 |
| 译文已就绪 | `TRANSLATION_READY` | 生成排版补丁 |
| 排版计划已就绪 | `PATCH_READY` | 渲染候选页 |
| 候选页已就绪 | `CANDIDATE_READY` | 执行裁决 |
| 质量裁决已完成 | `QUALITY_DECIDED` | 通过、修复或失败 |
| 局部修复中 | `REPAIRING` | 重新渲染和完整复裁 |
| 页面通过 | `PAGE_PASSED` | 进入文档合并 |
| 能力不足 | `CAPABILITY_FAILED` | 停止并记录缺失能力 |
| 产品质量失败 | `QUALITY_FAILED` | 停止或回到工具箱研发 |
| 流程契约失败 | `PROCESS_FAILED` | 停止并记录状态、合同或证据失真 |

## 6. P1——穿刺运行基建

### 6.1 阶段目标

建立所有工具箱都会依赖的最小运行骨架，但不实现任何分类专用排版算法。

### 6.2 任务

1. 建立项目入口、配置和测试目录；
2. 建立样本快照器和 `SampleManifest`；
3. 冻结 §2.4 最小合同及 Schema 校验；
4. 建立中文状态机和合法状态转换；
5. 建立 Run 目录、Artifact 索引和失败恢复记录；
6. 建立最小 `TranslationProvider`：一页一次调用千问；
7. 建立固定译文 Provider，用于排版确定性回归；
8. 建立 API Key 环境变量读取，禁止写入 Artifact；
9. 建立版本记录：分类树、合同、工具、Prompt、模型和参数；
10. 建立过程结论与产品结论两个独立输出。

### 6.3 最小纵向切片

```text
复制一个单页样本
  → 写入 SampleManifest
  → 构造 PageTranslationRequest
  → 固定译文 Provider / 千问 Provider 返回 PageTranslationBundle
  → 写入 RunManifest
  → 状态完整到“译文已就绪”
```

本阶段不生成排版候选 PDF。

### 6.4 Gate P1

- [x] 上游 PDF 哈希在运行前后不变；
- [x] 单页快照可追溯到原文档和原页码；
- [x] 请求和返回 ID 一一对应；
- [x] 固定译文 Provider 同一输入产生确定性输出；
- [x] 千问 Provider 完成至少一页真实调用；
- [x] 密钥不出现在代码、日志和报告；
- [x] 非法状态转换被拒绝；
- [x] 失败运行保留明确终止状态和错误证据。

### 6.5 进入 P2 的条件

Gate P1 全部通过，且合同没有依赖某个具体 leaf 的字段。否则先修正合同，不得让分类工具箱绕过公共 Runner。

任务：识别多个相对独立的文字块，保持每块左上角锚点和局部边界，禁止块间串写。

Gate：每块独立回填、块顺序不变、局部修复不影响其他块、regression/holdout 通过。

### 17.2 `body.flow_text.visual_anchored`

任务：处理围绕不可动视觉对象排列的正文流，固定视觉锚点，限制文字在自己的可用区域内适配。

Gate：视觉对象不动、正文不覆盖图片、阅读顺序不变、regression/holdout 通过。

### 17.3 Gate P11

- [ ] 先完成 `body.anchored_blocks` 的独立验收包；
- [ ] 再完成 `body.flow_text.visual_anchored` 的独立验收包；
- [ ] 两类页面均无块间串写或文字覆盖视觉对象；
- [ ] 任一公共内核修改已回归所有已晋级工具箱；
- [ ] 两个工具箱分别晋级或明确记录证据不足。

### 17.4 进入 P12 的条件

两个工具箱依次建设并分别形成独立 Gate。完成两份验收包后进入 P12；任一证据不足时必须记录缺口，依赖该能力的 composite 不得晋级。

## 18. P12——`chart` 与 `diagram`

### 18.1 共同边界

图片化图表或示意图内部文字不翻译、不重排。仅处理 PDF 中可定位、独立存在的原生标题、图例、注释或说明文字。

### 18.2 `body.chart`

任务：区分图表图像与原生文字对象；保持图表、坐标、颜色和数据视觉不变；仅回填允许的原生文字容器。

Gate：图表对象不变、数据标记不变、原生文字回填不覆盖图表、图片内文字未修改。

### 18.3 `body.diagram`

任务：区分示意图图像、矢量图形和独立原生文字；保持节点、连线和图形位置；仅处理允许的原生文字。

Gate：图形、节点和连线不变；译文不破坏图形层级；图片内文字未修改。

### 18.4 Gate P12

- [ ] 先完成 `body.chart` 的独立验收包；
- [ ] 再完成 `body.diagram` 的独立验收包；
- [ ] 每个工具箱分别完成 regression 和 holdout；
- [ ] 图片化内容没有被翻译或重排；
- [ ] 原生文字与视觉对象边界没有互相侵入；
- [ ] 样本不足时标记“证据不足”，没有为了后续 composite 强行晋级。

### 18.5 进入 P13 的条件

进入 P13 的条件：chart 和 diagram 已分别完成 Stage Gate；只有已晋级的基础能力才能被对应 composite 使用。

## 19. P13——其余 composite

按照下表顺序逐个建设，不一次性实现：

| composite | 前置工具箱 | 专用 Gate |
|---|---|---|
| `anchored_blocks_chart` | anchored_blocks、chart | 块修复和图表边界互不侵入 |
| `chart_table` | chart、table | 图表不可动，表格行列不变，区域不碰撞 |
| `flow_text_chart` | flow_text、chart | 文字流不覆盖图表，图表不参与文字重排 |
| `flow_text_diagram` | flow_text、diagram | 阅读顺序与图形结构同时保持 |

每个 composite 都要建立独立目录、样本、流程文档、Finding 和 `PromotionManifest`。基础工具箱已晋级不代表 composite 自动通过。

### 19.1 Gate P13

- [ ] 四个 composite 按表中顺序分别建立样本和流程文档；
- [ ] 每个 composite 分别完成固定译文、千问端到端、regression 和 holdout；
- [ ] 每个 composite 分别生成 `stage_gate.json`；
- [ ] 通过项分别生成 `PromotionManifest`；
- [ ] 未通过的 leaf 保持不支持，没有影响已晋级工具箱。

### 19.2 进入 P14 的条件

进入 P14 的条件：四个 composite 已按表中顺序完成 Gate；未晋级项已经写入能力矩阵，且不存在会影响核心工具箱的公共内核缺陷。

## 20. P14——工具箱目录与多页文档集成

### 20.1 阶段目标

将已晋级工具箱注册到文档级引擎骨架，验证真实多页 PDF 的拆分、分类结果导入、逐页处理、样式记忆和原序合并。

### 20.2 任务

1. 建立 `PageToolboxCatalog`；
2. 固定兼容元组：分类树版本、合同版本、toolbox_key 和工具箱版本；
3. 建立原文档级 `DocumentManifest`；
4. 拆页时保存页码、页框、旋转和原始哈希；
5. 建立只在同一原文档内有效的 `DocumentStyleMemory`；
6. 每页独立调用千问并运行对应工具箱；
7. 显式处理 `visual_only`、`freeform` 和未晋级 leaf；
8. 仅当所有必需页面通过时按原序合并；
9. 输出文档级过程结论和产品结论。

### 20.3 Gate P14

- [ ] leaf_key 只能调用对应已晋级工具箱；
- [ ] 版本不兼容时显式拒绝；
- [ ] 不同文档不共享样式记忆；
- [ ] 每页翻译请求独立；
- [ ] 合并后页数、页序、页框和旋转保持；
- [ ] 源 PDF 全程不修改；
- [ ] 任一必需页失败时文档不能伪装完成；
- [ ] 至少一个包含多类页面的代表性 PDF 完成真实端到端验证；
- [ ] 能定位最终问题属于哪一页、哪个 leaf、哪个工具和哪个 Finding。

### 20.4 进入 P15 的条件

核心工具箱已晋级，代表性多页文档完成拆分—逐页翻译—排版—裁决—合并闭环，且没有万能 fallback。

## 21. P15——全量回归与正式引擎移交

### 21.1 阶段目标

证明各工具箱稳定组合后没有相互污染，并形成正式引擎可以直接消费的合同、目录和证据。

### 21.2 测试矩阵

- 每个已晋级 leaf 的 development、regression、holdout；
- 每个历史失败样本；
- 非目标 leaf 哨兵样本，防止错误工具箱接管；
- 固定译文排版回归；
- 千问页级端到端回归；
- 多页拆分和合并回归；
- 字体、字形、不可动对象和源文件哈希回归；
- 修复接受、回滚、预算耗尽和无改善终止；
- 分类树/合同/工具箱版本不兼容测试；
- 能力不足和产品质量失败终态测试。

### 21.3 移交产物

- 支持能力矩阵；
- 已晋级工具箱版本清单；
- 未支持 leaf 和缺失证据清单；

- 千问适配器替换说明；
- 正式引擎装配说明；
- 全量回归报告；

### 21.4 Gate P15

- [ ] 所有已晋级工具箱独立回归通过；
- [ ] 非目标工具箱不会错误接管页面；

P15 Gate 通过后，才进入正式引擎工程建设或合并阶段。

## 22. 统一质量与晋级规则

### 22.1 硬门禁

任何工具箱晋级都必须满足：

- 源 PDF 不修改；
- 图片、颜色、左上角锚点和阅读顺序保持；
- 表格行列和单元格归属保持；
- 图片内文字不翻译、不重排；
- 无不可接受的越界、溢出、遮挡、串栏和错格；
- 字体有目标字形并正确嵌入；
- PageTranslationBundle 与文字容器 ID 完整对齐；
- 失败和能力不足有明确终态。

### 22.2 修复接受规则

一次修复只有同时满足以下条件才接受：

1. 目标 Finding 明确改善；
2. 没有新增硬约束 Finding；
3. 不可动对象仍保持；
4. 完整复裁通过。

否则回滚。达到最大轮次或连续无改善后停止，不允许无限修复。

### 22.3 固定译文与真实千问分轨

- 固定译文回归用于判断排版工具是否确定、稳定；
- 千问端到端用于判断真实页级翻译接口和工程回填是否成立；
- 两类证据缺一不可；
- 不允许把千问本次译文变化误判为排版算法回退。

### 22.4 过程结论与产品结论

- 过程结论：状态、工具、合同、Artifact 是否按要求执行；
- 产品结论：最终页面的翻译和排版是否真正可接受。

生成候选 PDF 只能证明流程走到候选阶段，不能证明产品通过。

## 23. 风险与处理

| 风险 | 影响 | 处理 |
|---|---|---|
| 过早抽公共能力 | 改一处影响多个工具箱 | 默认私有，跨两个 leaf 验证后才下沉 |
| 千问译文波动 | 无法判断排版回退 | 固定译文与真实千问分轨 |
| rare leaf 样本不足 | 虚假稳定 | 标记证据不足，不晋级 |
| 单页样本丢失文档上下文 | 字体和样式不一致 | 保存文档血缘，P14 验证 DocumentStyleMemory |
| Codex 长期充当自由引擎 | 运行不可复现 | 晋级前固化确定性流程 |
| composite 变成万能编排器 | 重走旧框架问题 | 仅允许有限、显式、leaf 私有组合 |
| 修复循环无边界 | 越修越坏或耗尽预算 | 最大轮次、无改善终止、完整复裁与回滚 |
| 字体缺字 | 乱码或视觉假通过 | 字形覆盖与字体嵌入硬门禁 |
| 密钥进入仓库 | 安全风险 | 环境变量，本地忽略，报告脱敏 |
| 候选存在即宣称成功 | 产品质量失真 | 过程与产品双裁决 |
