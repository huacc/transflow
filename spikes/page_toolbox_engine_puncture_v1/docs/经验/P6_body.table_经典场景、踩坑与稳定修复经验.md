# P6 `body.table` 经典场景、踩坑与稳定修复经验

## 1. 文档目的

本文只沉淀 `body.table` 穿刺中已经遇到、能够迁移到其他表格页的经典问题。每条经验都从具体场景出发，说明错误表现、错误假设、根因和通用解法。

运行编号、样本结论、测试数量、冻结状态和阶段 Gate 属于执行记录，不写入本文。样本文字、公司名、数字、页码和绝对坐标只能作为回归证据，不能成为运行时规则。

## 2. 元思想：PDF 绘制碎片不等于表格业务结构

PDF 提供的是文字和图形的绘制结果，不直接提供可靠的逻辑行、逻辑单元格或可写区域。稳定处理表格必须完成三步：

```text
从绘制碎片恢复逻辑所有权
→ 让目标语言在真实结构边界内自然排版
→ 用实际渲染字形验收
```

后续五类场景都来自同一个错误：把 PDF 抽取结果或规划结果，当成了最终业务结构和视觉事实。

## 3. 经典场景一：视觉行不等于逻辑单元格

**可复用场景**

- 一个长表格标签在 PDF 中被拆成多条视觉行；
- 金额、注号或比较期数值只出现在标签最后一条视觉行；
- 标签后续行有缩进，但仍属于同一个第一列单元格；
- 同一逻辑单元格的多行文字被抽取成多个 text block。

**踩过的坑**

- 把每条视觉行直接建成一条逻辑表格行；
- 前几行只有一个文字片段，便被误认为横跨整张表的合并格；
- 最后一小段文字被拆成独立单元格，翻译后形成孤立词；
- 最后一行同时存在粗体金额，整行字重因此被误判，续行合并失败。

**错误假设**

错误假设是“PDF 中同一 y 带上的对象构成一条业务行”。实际上，表格标签可以纵向跨越多个视觉行，而其他列只在最后一条基线上出现值。

**通用解法**

先在同一列内判断文字延续，再恢复逻辑单元格：

- 列归属相同；
- 左锚点相同或只有合理的层级缩进；
- 字号、颜色和当前文字列自身的字重兼容；
- 纵向间距属于当前表格的连续行节奏；
- 前一片段没有明确结束；
- 中间没有真实横线、分节标题或其他结构边界。

判断样式时只读取当前文字列，不能把同行金额列的粗体、颜色或对齐混入标签证据。合并成功后只扩大纵向 `row_span`；没有正向跨列证据时，`column_span` 仍保持原列。

**一句话原则**

> 表格中的换行通常是同一个格的纵向延续，不是新业务行，更不是跨列证据。

## 4. 经典场景二：目标语言放不下时，横向守列、纵向适配

**可复用场景**

- 文字标签旁边紧邻金额、百分比或比较期列；
- 目标语言比源语言更长，需要增加换行；
- 窄表头、分节标题或多级缩进标签在原字号下无法容纳；
- 页面下方仍有纵向空间，但水平方向没有安全空白。

**踩过的坑**

- 因为当前视觉行只有一个标签，就把文字格扩到整表宽度；
- 借用相邻金额列，让译文覆盖数字或底色列；
- 为了保持单行而无限缩小字号；
- 只向下延伸标签文字，却不处理同一行的数值、横线和后续行。

**错误假设**

错误假设是“放不下说明文字框应该变宽”。表格列不仅是几何区域，也是语义所有权；侵入相邻列会让标签和数值的对应关系失真。

**通用解法**

采用固定顺序消化语言长度变化：

```text
确认逻辑单元格归属
→ 严格保持所属列宽
→ 在列内按目标字体自然换行
→ 利用单元格已有上下留白
→ 调整字号和行距
→ 页面确有安全空间时再考虑整行增高
→ 仍无法容纳则明确失败
```

若允许纵向扩行，必须把它作为整行结构变化处理：同行保护数值、后续行、横线、竖线、底色和表格下方内容必须同步移动。只移动标签会破坏行对应关系。

**一句话原则**

> 表格翻译应横向守列、纵向适配，不能用侵入邻列换取文字 fit。

## 5. 经典场景三：文本框 fit 不等于真实字形没有压线

**可复用场景**

- 排版器返回文本已经放入 bbox，但候选图中横线穿过第二行文字；
- 单元格边界由相邻文字中点推断，而 PDF 中存在位置不同的真实横线；
- 英文小写下伸部、粗体字形或较紧行距让文字比预想更靠近边框；
- 表格使用分段横线、描边矩形或填色列，纯文本几何无法表达真实边界。

**踩过的坑**

- 只检查输出 bbox 是否位于推断单元格内；
- 把相邻文字的中点当成真实行边界；
- 把 `insert_textbox` 的成功返回值当成视觉 PASS；
- 只看规划框，不检查渲染后的实际字形范围。

**错误假设**

错误假设是“规划几何安全就等于视觉安全”。文本框描述的是允许排版的区域，实际字形还受到字体 ascent、descent、字重和行距影响；推断边界也可能与真实矢量线不同。

**通用解法**

表格边界采用以下证据优先级：

```text
位于文字间隙中的真实矢量线
> 原生文字对象边界
> 稳定文字间隙
> 几何中点启发式
```

规划时从真实横线、竖线向格内留出与字号相关的安全距离。候选生成后重新提取实际字形 bbox，并逐条检查它是否与保留的表格线相交。任一相交都属于产品失败，即使文本框 fit、PDF 可打开、其他对象哈希正常。

**一句话原则**

> 规划阶段看真实结构边界，验收阶段看实际字形；容器安全不代表视觉安全。

## 6. 经典场景四：同一视觉区域不代表具有相同可编辑性

**可复用场景**

- 页底同一行同时出现公司名和页码；
- 表格辅助行同时出现文字标签、注号和金额；
- 一个单元格中同时包含说明文字、日期、币种或标准编号；
- 表下注释附近还有独立的页面 footer。

**踩过的坑**

- 把整个 footer 视为保护区域，导致原生公司名漏译；
- 把整条辅助行送去翻译，导致金额、页码或注号被改写；
- 仅按视觉区域决定“整块翻译”或“整块保护”；
- 为保护一个数字而放弃翻译同格中的说明文字。

**错误假设**

错误假设是“位于同一区域的对象具有相同操作属性”。区域只能说明对象属于页面的哪个部分，不能说明每个对象是否应该翻译。

**通用解法**

先按水平间距、对象边界和内容类型拆分操作单元：

- 公司名、文字标签和说明句进入翻译单元；
- 纯页码、金额、百分比、币种、日期、注号和占位符保持原样；
- 混合文字格允许翻译，但受保护 token 必须在目标文本中保持等价；
- 表下注释与页面 footer 通过相对页高、间距和对象簇区分，不能依赖固定措辞。

**一句话原则**

> 区域归属决定对象属于哪里，对象类型才决定它能不能翻译。

## 7. 经典场景五：PDF 文字对象不等于肉眼看到的一份文字

**可复用场景**

- 同一标题在近似 bbox 被重复绘制；
- 两个副本颜色、字体或字重略有不同；
- PDF 为阴影、印刷层、可访问性层或导出兼容性保存了重叠文字；
- 抽取器返回多个对象，但页面上只看到一份文字。

**踩过的坑**

- 把每个 text object 都建成独立翻译单元；
- 只擦除其中一个对象，候选中残留幽灵原文；
- 使用先绘制副本的样式，结果与肉眼看到的顶层样式不一致；
- 去重时只保留一个对象 ID，渲染阶段无法完整删除原文。

**错误假设**

错误假设是“一个 PDF 文字对象对应一份可见文字”。实际上，肉眼看到的是按绘制顺序叠加后的结果。

**通用解法**

对文字相同、bbox 高度重合且坐标差位于字体尺度容差内的对象建立重叠绘制组：

- 模板只生成一个逻辑翻译单元；
- 可见样式采用最后绘制、位于视觉顶层的对象；
- 翻译单元保存该组全部源对象 ID；
- 擦除时删除全部副本，再只回填一次译文。

**一句话原则**

> 建模板时要恢复可见文字，不能机械地把每个 PDF text object 都当成独立内容。

## 8. 五类经验的共同判断法

遇到新的表格排版问题时，依次问三个问题：

1. 这批 PDF 绘制碎片在业务上属于哪个逻辑单元格；
2. 该单元格真正不能跨越的列、矢量线和保护对象是什么；
3. 候选渲染后的实际字形是否仍满足这些边界。

如果第一个问题没有回答清楚，调字号和 bbox 只会掩盖模板错误；如果第二个问题依赖推断而忽略直接矢量证据，所谓“格内写入”可能是假的；如果第三个问题只看规划框而不看真实字形，机械 PASS 仍可能是视觉 FAIL。
