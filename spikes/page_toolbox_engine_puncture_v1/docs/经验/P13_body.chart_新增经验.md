# P13 `body.chart` 新增经验

## 1. 收录边界

本文只记录 `body.chart` 独立穿刺中新增或被进一步明确的通用经验，不重复 P4 正文流、P6 表格、P11 独立锚定块和 P14 示意图已经完整讨论的内容。

本文不记录样本 ID、公司名、页码、具体标题、年份、绝对坐标或某次模型返回文本，也不把当前样本池中的偶然尺寸写成运行时规则。

当前证据能够说明这些规则在已知图表页池和统一尺度变换下保持稳定，但已预览并参与修复反馈的样本只能作为回归证据，不能继续宣称严格盲测晋级。

## 2. 总体认识：图表页排版不是自由正文流，也不是原框覆写

图表页同时包含三类对象：

1. 锁定视觉对象：柱、线、点、扇区、环、坐标轴、图例色块、图片、背景和装饰；
2. 保护文字：页码、纯数值、刻度、数据标签、标准单位和不应翻译的结构记号；
3. 语义文字：页眉页脚中的自然语言、图表标题、图例名称、轴/类别标签、注释和局部表格文字。

第一类不能移动、缩放或重绘；第二类不能被翻译覆盖；第三类需要翻译，但不能失去与图形、列、色块或页边的原始语义关系。

稳定的处理顺序应是：

```text
页面事实
-> 视觉区域和锁定对象
-> 每个原生文字对象的唯一 owner
-> 语义角色
-> 图形关联与左/右/中心锚点
-> 页面、邻居和锁定对象形成的安全区域
-> 同类字体比例和有限试排
-> 删除原文并回填译文
-> 实际字形、图形碰撞和语义关联复裁
```

一句话原则：

> 先确定文字属于谁、与谁对齐，再决定译文怎样装入；不能先塞字，再倒推语义关系。

## 3. 新经验一：页眉页脚按语义翻译，不能按页面位置一刀切

### 场景

年报页眉页脚常把公司名、报告名称、章节名、年份、页码、Logo 和装饰线拆成多个 PDF 对象。它们处在同一个 margin 区域，但可编辑性并不相同。

### 容易出现的误判

- 因为文字位于页眉或页脚，就把整行全部保护而不翻译；
- 把页码和公司名合成一个翻译容器；
- 只给短中文的源字形窄框，导致英文公司名或报告名被异常断词；
- 为了容纳译文移动页码、Logo 或装饰线；
- 竖排页脚翻译后丢失旋转方向。

### 新认识

`PAGE_HEADER` 和 `PAGE_FOOTER` 是位置及排版角色，不等于“禁止翻译”。还需要在角色内部继续判断语义：

| 内容 | 处理 |
|---|---|
| 公司名、报告名、章节名等自然语言 | 进入翻译请求 |
| 页码、纯数字、独立年份 | 保护或作为必保留字面量 |
| Logo、色块、分隔线和装饰 | 锁定 |
| 自然语言与年份混排 | 翻译自然语言，保持年份字面量和顺序 |

同一 margin 行应由锁定对象切分成互不连通的安全横向区段。可翻译片段在所属区段内协同使用空间，保持原阅读顺序和页边锚点；不能让每个片段各自扩宽后互相覆盖。

左侧页眉通常保持左锚，右侧页脚通常保持右锚，居中页眉保持原中心轴。竖排页脚继续保留原旋转方向。源字形 bbox 只提供起点和基线，不是目标语言可用宽度。

一句话原则：

> 页眉页脚的“家具”要锁定，页眉页脚的“语言”仍要翻译。

## 4. 新经验二：源字形 bbox 是锚点事实，不是译文牢笼

### 场景

中文标签可能只有几个字，源 bbox 很窄；翻成英文后变成多个单词。页面横向仍有大量空白，但如果强制限制在源 bbox 内，就会产生极小字号、逐词断裂或错误的“不翻译”。

### 容易出现的误判

- 把源文字的 `x0`、`x1`、宽度和高度全部视为不可改变；
- 只因为英文超出源字形框就判定布局失败；
- 给所有文字统一扩大到整页宽度，破坏图文关系；
- 居中文字扩框后仍按左对齐绘制，视觉中心漂移；
- 右侧标签向右扩展进入图形，而不是向外侧使用空白。

### 新认识

跨语言排版真正需要保存的是语义锚点：

- 左对齐文字保持源 `x0`；
- 右对齐文字保持源 `x1`；
- 居中文字保持源中心轴；
- 与图例色块、圆环、柱图或轴线关联的标签保持相对方向；
- 图片和图形本身保持锁定。

源 bbox 是最紧的第一试排槽位。若译文放不下，应继续尝试同一语义带内的安全横向空间和纵向空间，直到页面边界、相邻文字或锁定视觉对象为止。

一句话原则：

> 跨语言保持的是边缘锚点、中心轴和图文关系，不是源字形的偶然宽度。

## 5. 新经验三：横向关系稳定，纵向布局可以依据留白弹性适配

### 场景

有些图表页正文或注释上下留白很大，但译文仍被压在源文字顶部的一条窄带中。机械上没有越界，视觉上却拥挤、字号偏小、行距过紧。

### 容易出现的误判

- 只检查左右边界，不利用真实存在的纵向留白；
- 为了“忠实坐标”让多行译文全部挤在源第一行附近；
- 一看到纵向空白就无上限放大字体；
- 每个段落独立放大，导致同级正文大小不一致；
- 为了舒展文字而改变图表、图片或页边家具的位置。

### 新认识

横向位置主要承载语义锚点，通常应稳定；纵向位置可以综合以下事实研判：

- 译文长度和预计行数；
- 源文字本来是单行还是多行；
- 上下安全空间；
- 相邻标题、正文、图表和页脚的位置；
- 同级文字的字号与行距；
- 页面边界和锁定图形。

单行文字优先保持源字号并扩展安全宽度；原本就是多行、且纵向空间充足的正文，才允许使用有限的源字号相对放大和更舒展行距。放大幅度必须有上界，并与同类正文统一。

译文无法保持单行时可以自然换成多行。换行本身不是失败；越界、碰图、破坏阅读顺序或把同级文字压成不同大小才是失败。

一句话原则：

> 让 bbox 服务于文字和版面关系，而不是让文字服从源字形紧框。

## 6. 新经验四：字号一致性必须按语义角色分组

### 场景

同一页面可能同时存在标题、正文、图例、类别标签、局部表格表头、表格正文和页眉页脚。译文长度不同，逐框独立 fit 很容易让同级文字出现多个字号。

### 容易出现的误判

- 长译文单独缩小，短译文保留原字号；
- 表格中的每个单元格独立求字号；
- 表头和表体混用一个缩放比例；
- 用文字字符数作为正文与短标签的分界；
- 因为某段文字恰好跨过一个长度阈值，就改变排版策略。

### 新认识

至少应分别维护以下字体组：

1. 页面或图表标题；
2. 普通正文和长注释；
3. 图例及轴/类别标签；
4. 局部表格表头；
5. 局部表格正文、分节和合计；
6. 页眉页脚。

同组文字使用一致的源字号比例和行距档位；不同组继续保留源页的层级差。组的形成依据角色、字体样式、源字号和视觉关系，不能依赖某个样本句子或任意字符数边界。

有限的固定修复步长是允许的，例如字号逐档变化或行距逐档变化，但它们只能是通用试排原子：

- 不因样本文字、年份或文件名触发；
- 不绑定某页绝对坐标；
- 有明确的最小可读字号和最大放大比例；
- 在不同页面尺寸和文字长度下接受变形测试；
- 同一语义组统一选择档位。

一句话原则：

> 可以固定“怎么逐步试”，不能固定“某一页最后必须是多少”。

## 7. 新经验五：图表侧标签应向远离关联图形的方向扩展

### 场景

饼图、环图、柱图和多面板图表的类别标签常位于视觉对象边缘。目标语言变长后，如果按普通左对齐文字向右扩框，可能直接进入圆环、柱体或图片。

### 容易出现的误判

- 所有左对齐文字都向右扩展；
- 根据具体标签文字决定扩展方向；
- 根据某页固定坐标判断“这是左侧标签”；
- 把铺满面板的背景误认为标签右侧的主体图形；
- 只检查允许框与图形相交，不检查实际字形。

### 新认识

扩展方向应由当前页的相对几何关系决定：

- 标签位于关联图形左侧时，优先保持右边缘并向左使用空白；
- 标签位于关联图形右侧时，优先保持左边缘并向右使用空白；
- 标签接近图形顶部或底部时，优先保持源水平锚点，避免误伸入相邻面板；
- 覆盖整个卡片的背景面板不能单独决定外扩方向；
- 最终碰撞应使用实际插入字形，而不是最大允许框。

方向判断只依赖标签与关联区域、锁定视觉对象和页面边界的相对关系，不能出现样本标题、数字或绝对坐标分支。

一句话原则：

> 图表外侧标签不是“向左或向右排”，而是“向远离关联图形的一侧排”。

## 8. 新经验六：图表页中的局部表格应借鉴 table 方法论，但不能越过叶子边界

### 场景

以图表为主体的页面底部或侧边可能附带一块结构化表格。若仍按普通注释合并，表格多行会被合成段落；若直接把整页交给 `body.table`，又会失去图表标签和视觉关联。

### 容易出现的误判

- 因为页面中出现表格，就把整页重分类为 `body.table`；
- 因为分类是 chart，就忽略局部行列结构；
- 按文本内容或表头词语识别表格；
- 直接跨叶子导入完整 table 运行时，形成隐式耦合；
- 表格正文、表头和合计使用不同的任意字号。

### 新认识

局部表格可以复用 P6 的方法论：

- 视觉行不等于逻辑单元格；
- 稳定基线、列锚点和重复行关系共同证明局部表格结构；
- 横向守列，纵向适配；
- 表头、表体、分节和合计分别形成字体组；
- 最终检查实际字形是否压线或跨列。

但复用的是判断思想和通用机械能力，不是让 `body.chart` 依赖 `body.table` 的样本规则或运行目录。

若图表和大表格都成为页面不可忽略的主体区域，应由 `body.composite.chart_table` 处理；只有从属于图表页的局部表格才留在 `body.chart`。

一句话原则：

> 局部表格要识别其行列结构，但不能因此丢掉整页的图表主语义。

## 9. 新经验七：翻译准确度可以放宽，翻译完整性不能放弃

### 场景

模型译文可能术语不够精准，但结构完整；也可能只返回一个很短的任意短语、问号、占位符、原文或遗漏部分容器。前者可以进入排版，后者不能。

### 容易出现的误判

- 用术语映射表绑死具体公司或具体句子；
- 因为译文不够专业就阻断布局穿刺；
- 因为任意短回复能够轻松装入，就当作有效翻译；
- 页眉页脚没有进入请求，最终保留源语言；
- 第一次必保留字面量失败后，重试结果不重新校验；
- 一个容器失败就重新请求整页，增加不必要波动。

### 新认识

P13 不负责术语精修，但必须保证翻译合同：

- 每个输入容器恰好有一个对应输出；
- 不允许占位符、问号式敷衍或原文原样返回；
- 长源文不能只返回明显不充分的极短译文；
- 中译英不得残留应翻译中文，英译中同样需要检查明显漏译；
- 页眉页脚中的自然语言与主体文字使用同一完整性门禁；
- 数字、年份和必要结构字面量必须保留；
- 只重试失败 ID，并在每次返回后重新执行完整门禁。

运行时不需要维护人工翻译映射。模型返回的真实译文只要满足完整性合同，就应进入布局和 PDF 回填。

一句话原则：

> 可以接受“不够精准”，不能接受“没有真正翻译”。

## 10. 新经验八：失败页面也必须产出可审阅 PDF

### 场景

批次中可能出现 Provider 失败、译文校验失败、字体缺字、布局溢出或图文碰撞。如果失败时不生成 PDF，只留下 JSON，使用者无法直接判断问题属于翻译、布局还是渲染。

### 容易出现的误判

- 只有 PASS 页面才输出 `candidate.pdf`；
- 有完整译文但布局失败时直接复制源页；
- 诊断 PDF 只回填成功容器，失败容器继续保留源文；
- 因为生成了 PDF，就把失败页面计为产品通过；
- 为了放下诊断译文扩展页面尺寸。

### 新认识

每个已建立运行包的页面都必须生成 `output/candidate.pdf`：

- 正常通过：输出正式候选；
- 已有完整译文但布局或质量失败：删除对应源文，尽量在原页面范围回填全部译文，输出明确失败的诊断候选；
- Provider 完全没有返回可用译文：允许源页兜底，但状态必须明确为能力失败；
- 所有候选保持单页和原页面尺寸；
- PDF 是否存在与 Product 是否 PASS 分开裁决。

失败清单至少记录样本、终态、Finding、owner、PDF 是否存在以及直接可见问题。

一句话原则：

> PDF 是问题证据，不是成功证明；失败也必须让人看得见。

## 11. 新经验九：类 Drools 规则应围绕事实、Finding 和修复原子分发

### 场景

图表页问题类型较多：源框过窄、纵向空间未利用、同组字号不一致、外侧标签方向错误、局部表格错组、页眉页脚漏译等。如果全部写进一个大函数或交给模型自由决定，规则适用域会快速失控。

### 新认识

每条规则应明确四部分：

1. 适用事实：角色、锚点、图形关联、相邻障碍、字体组和可用空间；
2. 失败类型：例如源框过窄、需要多行、同组字号不一致或实际字形碰图；
3. 修复原子：安全横向扩展、安全多行换行、组内字号档位、向外侧扩展或诊断物化；
4. 复裁工具：实际字形、页面边界、图形签名、保护文字和翻译完整性。

规则只负责确定可重复的布局动作，模型主要负责翻译。规则无法在合同内解决时应诚实失败，不能临时增加样本分支。

规则轨迹应能回答：使用了哪些事实、命中了什么问题、选择了哪个修复原子、为什么没有调用其他工具。

一句话原则：

> 规则分发的是有限动作和证据，不是把整页交给一个不可解释的自由裁决。

## 12. 新经验十：反过拟合审计必须同时覆盖身份、Prompt 和结构变形

### 场景

运行时代码中搜不到样本 ID、公司名或固定句子，并不代表没有过拟合。规则仍可能依赖当前样本共有的页面尺寸、字体点值、文字长度或图形数量。

### 新认识

P13 至少需要三层反过拟合审计：

### 12.1 身份扫描

- 运行时代码中无样本 ID、文件名、公司名、标题和年份分支；
- 无已知页绝对坐标四元组；
- 无跨叶子样本夹具进入运行时。

### 12.2 Prompt 扫描

- Prompt 也属于运行时代码；
- 不写已知样本句子或翻译答案；
- 不用具体示例暗示某一页的标签方向和布局；
- 必保留标记使用抽象合同说明。

### 12.3 结构变形探针

- 页面、字体、文字和视觉对象统一缩放；
- 在不改变角色的情况下改变文案长度；
- 比较长度边界两侧的角色和字体策略；
- 改变图表尺寸和空白比例但保持相对拓扑；
- 检查容器 owner、角色、对齐、锚点和视觉区域是否稳定。

固定值不是一律禁止。可接受的是通用机械步长，例如字号每档变化、行距每档变化和最小可见间隙；不可接受的是用固定值表达某个样本事实，例如特定文字、特定页坐标或特定图形数量。

判断固定值是否合理，可以问三个问题：

1. 它是否会因更换文字内容而改变规则分支；
2. 它是否随字体、页面或关联区域尺度进行归一化；
3. 它是否经过缩放和长度边界探针。

一句话原则：

> 搜不到样本身份只是第一层；经过结构变形仍保持结论，才有更强的泛化证据。

## 13. 新经验十一：机械门禁与视觉审美必须分层验收

### 场景

页面可能满足 bbox 不越界、PDF 可打开、图形哈希不变等机械条件，但仍存在文字过密、行距不自然、同级字号不一致、标签视觉偏心或表格列对不齐。

### 新认识

验收应按递进节奏执行：

1. 先跑一页，定位一个主要病因；
2. 再跑二到三页不同形态，确认规则不是只修一页；
3. 最后运行全部样本，每页无论成功或失败都产出 PDF；
4. 对全量候选生成缩略图总览，检查整页崩坏、越界和异常密度；
5. 对长正文、页眉页脚、局部表格、环图侧标签和竖排文字等重点页放大复核；
6. 机械裁决使用实际字形 bbox，而不是只使用最大允许槽位。

页面存在留白时，应判断文字是否可以更自然地使用空间；但视觉优化仍受源字号比例、同类一致性、图文关系和页面边界约束，不能以“更美观”为理由任意重排。

一句话原则：

> 机械 PASS 证明没有触犯合同，视觉复核才证明页面像一份正常文档。

## 14. P13 新增经验检查表

### 14.1 翻译边界

1. 页眉页脚中的自然语言是否进入翻译请求；
2. 页码、纯数值、单位、Logo 和装饰是否继续保护；
3. 是否存在原文原样返回、占位符、任意短回复或源语言残留；
4. 必保留字面量失败重试后是否重新校验。

### 14.2 owner 与锚点

1. 每个原生文字对象是否恰有一个 owner；
2. 左、右、中心对齐是否保存对应边缘或中心轴；
3. 图例和类别标签是否仍与原图形、色块和列保持关系；
4. 页眉页脚是否在自己的 margin 安全带内协同布局。

### 14.3 字体与空间

1. 普通正文、表格正文和表头是否分别保持组内一致；
2. 是否先利用安全宽度和纵向留白，再降低字号；
3. 原本多行且空间充足的正文是否可以有限舒展；
4. 是否出现由任意字符数阈值导致的字号跳变；
5. 所有字体是否满足最小可读字号和嵌入要求。

### 14.4 图表与局部表格

1. 外侧长标签是否向远离关联图形的一侧扩展；
2. 顶部、底部叠加标签是否误伸入相邻面板；
3. 局部表格是否保持列锚点、表头和表体角色；
4. 图表和大表格同时主导时是否转交 composite 叶子。

### 14.5 失败证据与反过拟合

1. 每个运行页面是否都有非空、可打开、尺寸不变的 `candidate.pdf`；
2. 失败 PDF 是否仍完整暴露已有译文和直接问题；
3. 运行时及 Prompt 是否无样本身份和翻译答案绑定；
4. 是否执行尺度、文案长度和相对几何变形探针；
5. 是否把已预览 holdout 诚实标记为非盲回归证据；
6. 是否完成单页、二到三页、全量总览和重点页放大复核。
