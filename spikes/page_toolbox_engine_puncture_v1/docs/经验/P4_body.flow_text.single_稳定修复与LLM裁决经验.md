# P4 `body.flow_text.single` 稳定修复与 LLM 裁决经验

## 1. 文档目的

本文沉淀 P3/P4 单列正文穿刺中已经验证有效的调试、裁决、修复和工程固化方法，供后续分类工具箱复用。

复用边界必须明确：后续工具箱可以复用运行包、状态机、证据合同、千问适配器、单病因裁决范式、分发表结构和修复循环；不得直接复制单列正文的栏道推导、页面阈值或页面知识。

表格专属的经典场景、踩坑和稳定解法见 [`P6 body.table` 经验](./P6_body.table_经典场景、踩坑与稳定修复经验.md)。

## 2. 当前事实与结论边界

- 第 33 轮对 `body.flow_text.single` 全量 278 页完成端到端执行；
- 256 页过程 PASS，255 页机械产品 PASS；
- 23 页未闭合，其中 22 页停在能力或流程边界，1 页为产品 FAIL；
- 用户已对代表性效果完成人工认可；
- 这些事实足以证明工程范式和主要排版方法可复用，但不能把未闭合页面写成 PASS，也不能据此伪造正式 `PromotionManifest`。

因此，后续 P5 可以复用 P4 经验开展开发穿刺；P4 是否正式晋级仍以独立 Gate 和完整证据包为准。

## 3. 元经验：一个新分类工具箱第一步怎么走

这一节是本文最重要的部分。它不是总结 P4 最后有哪些工具，而是告诉下一个 Codex：拿到一个新分类时，如何从零启动，避免先写一堆后来用不上的代码。

### 3.1 不要从“复制上一个工具箱”开始

新分类的第一步不是复制 `single/tools/`，也不是先设计完整状态机。先回答三个问题：

1. 这个分类与上一个分类真正相同的工程环节是什么；
2. 这个分类新增的版式不变量是什么；
3. 第一页上最可能失败的是翻译、模板、布局、渲染还是裁决。

只有与页面类型无关的能力才直接复用，例如 PDF 事实提取、千问接口、运行包、源文件哈希保护、字体探测和过程/产品双裁决。单列的正文流、段落合并、栏道推导和修复规则不能因为代码已经存在就直接套到多列、表格或图表页面。

### 3.2 先读四类材料，不要盲写代码

Codex 开始一个新工具箱前，按顺序读取：

1. 当前建设计划中该阶段的任务、Gate 和进入下一阶段的条件；
2. 分类引擎给出的分类定义、正例、反例以及实际样本目录；
3. 上一个已验证工具箱的运行包、失败样本和最终代码；
4. `pdf_translation_workflow_core` 与 `pdf_translation_workflow_lab` 中与当前病因直接相关的合同和工具。

目的不是把旧项目全部搬过来，而是建立一张最小复用清单：

| 结论 | 动作 |
|---|---|
| 页面类型无关且已经稳定 | 直接 import 或抽到公共模块 |
| 方法论相同但页面知识不同 | 复用合同/接口，重新实现分类专用工具 |
| 旧工具只在某轮样本有效 | 只作为失败经验，不复制代码 |
| 当前分类暂时没有证据需要 | 不实现，记录为候选能力 |

### 3.3 第一页怎么选

第一张开发页应满足：

- 属于当前分类的清晰正例；
- 主结构明确，最好只有一个主要版式难点；
- 能覆盖本分类的核心不变量；
- 不来自 holdout；
- 不是最简单的空白页，也不是同时包含五六种结构的最难页。

例如单列正文的第一页应有标题、正文段落和明确的纵向阅读流；多列正文的第一页应先选栏边界清楚的两列页，而不是直接选跨栏标题、三列、侧栏和图表混在一起的页面。

### 3.4 第一页先做基线，不要先修

固定第一页后，先生成一个完整基线运行包：

```text
input/source.pdf
→ page_facts.json
→ page_template.json
→ translation_request.json
→ translation_bundle.raw.json
→ layout_plan.json
→ candidate.pdf
→ source/candidate/comparison.png
→ quality_decision.json
→ state_trace.json
```

即使结果很差，也先保留。没有基线就无法证明后续修改解决了什么，也无法判断是否引入回归。

### 3.5 第一次对比按五层排查

Codex 同时看原文 PDF、候选 PDF、并排图和结构化证据，依次判断：

1. **翻译层**：译文是否真实、完整、按容器 ID 返回，是否重复或串块；
2. **模板层**：语义段落、标题、正文、列表、脚注和控件是否分对，阅读顺序是否正确；
3. **布局层**：x0、宽度、字号层级、行距、段距和纵向流是否合理；
4. **渲染层**：原文是否擦净，图片/背景/颜色是否保持，字体是否嵌入；
5. **裁决层**：机械 PASS 是否与肉眼一致，已有 Finding 是否漏判或误判。

必须先确定问题所有者，再改代码。模板分错时调字号没有意义；译文重复时调 bbox 也没有意义；候选明显难看但机械 PASS 时，应先补裁决能力，而不是扩大样本。

### 3.6 第一轮只解决一个病因

选当前页最主要、最靠前的根因：

```text
模板错误优先于布局参数
→ 越界/重叠优先于视觉节奏
→ 局部明确病因优先于全页“看起来不好”
```

只改该病因的拥有模块，只调用一个修复原子，重新运行同一页。当前页从统一引擎入口稳定复现后，才换下一页。

### 3.7 每修好一页都要完成“工程固化”

人工把某个 run 的 PDF 修好不算完成。Codex 必须同步完成：

- 将病因加入注册表或工具箱分发表；
- 将判断固化为确定性规则或单病因千问 Prompt；
- 将修复写成独立工具/RepairPatch，而不是运行目录里的临时改图；
- 在统一引擎入口接入；
- 增加中文注释和回归测试；
- 重跑刚才那一页，证明不需要 Codex 再临场干预；
- 再挑结构差异明显的下一页。

这就是“先修一页”的真正含义：不是只做一页，而是用一页建立该分类的第一条可重复工程能力。

## 4. Codex 作为探索引擎时具体怎么工作

### 4.1 四个角色的职责

| 角色 | 在 P4 中的职责 | 不应承担的职责 |
|---|---|---|
| Codex | 读取证据、比较原候选、提出病因、定位拥有模块、设计规则/Prompt/工具、修改代码、重跑并最终复核 | 长期依赖临场手改候选 PDF |
| 工程引擎 | 按固定状态机采集、翻译、规划、渲染、规则裁决、静态分发和记录证据 | 自由发明病因或任意调用工具 |
| 千问 | 页级翻译；对工程证据无法确定的单一视觉维度作结构化裁决 | 管整个工作流、一次判断所有问题、输出任意工具名 |
| 用户/人工 | 对代表页作最终美观度签收，指出工程指标尚未表达的视觉问题 | 替代引擎生成过程证据 |

早期由 Codex 充当“元引擎”：它观察自己为了修对一页调用了什么事实、作了什么判断、选择了什么工具。每次闭环后，把这些判断写入规则、Prompt、分发表和引擎。随着固化增加，Codex 的临场工作逐步减少，工程引擎承担正常执行。

### 4.2 Codex 的完整诊断循环

```text
读取分类定义和当前页
→ 保存源页基线
→ 执行当前工程引擎
→ 同时检查 PDF、并排图、PageFacts、模板、译文和布局计划
→ 列出候选病因，但只选择一个主病因
→ 判断是否有可直接计算的证据
→ 有：写确定性规则；无：写单病因千问 Prompt 并请求裁决
→ 将结论规范化为 failure_class
→ 查静态 failure_dispatch_table
→ 选择已注册 repair_atom 和 bound_tool
→ 生成并应用一个 RepairPatch
→ 重新渲染、重新采集、重新裁决
→ 目标改善且其他硬门禁不退化才接受
→ 将有效逻辑接回统一引擎并增加回归测试
```

### 4.3 Codex 如何判断应该改哪一层

| 现象 | 优先检查 | 代码拥有者 |
|---|---|---|
| 容器缺失、两段合并、同段拆散、阅读顺序错 | `page_template.json` 与源对象 | `PageTemplateBuilder` |
| 译文缺失、重复、ID/顺序错、元描述式伪翻译 | raw/bundle/translation trace | translation provider 与规范化层 |
| x0/宽度漂移、纵向越界、字号过小、段距异常 | `layout_plan.json` 与源 bbox | `LayoutPlanner` / RepairPolicy |
| 图片背景改变、选框漂移、擦除残留、字体未嵌入 | render evidence 与裁剪图 | PDF renderer / candidate repair |
| 肉眼明显错误但自动 PASS | Finding、Prompt 与 Gate 缺失 | Judge / validator |
| 修复目标改善但新重叠增加 | before/after quality signals | Repair acceptance，应回滚 |

### 4.4 Codex 怎样修改，而不是只手工调 PDF

1. 在当前 run 中记录病因证据和最小修复目标；
2. 修改拥有该病因的最小模块，不顺手重构相邻代码；
3. 工具参数来自当前页事实，不写样本专属常量；
4. 关键代码增加中文注释，说明规则为什么存在；
5. 生成 RepairPatch，保存 before/after；
6. 从正式 `run_p4_page` 入口重跑当前页；
7. 回放历史页，确认之前修好的问题没有复发；
8. 最后才扩大样本。

如果只在 `runs/<run>/output/candidate.pdf` 上手工修补，而没有修改引擎，经验没有沉淀，下一轮一定会复发。

## 5. 从 `pdf_translation_workflow_core` 抽取了什么

P4 没有复制整个 core，而是把 core 当作方法论和合同仓库，按单列正文需要抽取最小子集。

### 5.1 状态机与诚实终态

来源：

- `pdf_translation_workflow_core/contracts/state_machine.md`；
- `pdf_translation_workflow_core/README.md`。

抽取经验：

- `PageFacts → PageTemplate → Translation → LayoutPlan → Candidate → Judge` 各阶段必须有可核对 Artifact；
- 修复计划不等于修复执行；进入修复循环必须生成 RepairPatch、重新生成候选并复判；
- 明确区分过程失败、产品失败和能力不足；
- `process_verdict` 与 `product_verdict` 分开记录。

落地位置：

- `toolboxes/body/flow_text/single/tools/p4_engine.py`；
- 页级 `reports/state_trace.json`、`quality_decision.json` 和 `run_result.json`。

### 5.2 病因注册表和修复原子

来源：

- `pdf_translation_workflow_core/contracts/registry/failure_classes.json`；
- `pdf_translation_workflow_core/contracts/registry/repair_atoms.json`；
- `pdf_translation_workflow_core/contracts/page_type_repair_matrix.md`。

抽取经验：

- 病因必须标准化，不能只说“不好看”；
- 一个病因对应允许的修复原子、目标状态和验证方式；
- 一轮只选一个主病因，其他病因延期；
- 页面类型和区域类型决定可用工具，不能所有页面共用一个万能修复器。

P4 实际只选取单列正文相关病因，例如：

- `paragraph_density_mismatch`；
- `internal_paragraph_gap`；
- `section_spacing_regression`；
- `font_hierarchy_ratio_mismatch`；
- `line_fragmentation`；
- `source_anchor_order_mismatch`；
- `icon_label_misalignment`。

落地位置：

- `toolboxes/body/flow_text/single/contracts/failure_dispatch_table.json`；
- `tools/validators/`、`tools/probes/` 和 `tools/repairs/`。

### 5.3 分层模型裁决合同

来源：

- `pdf_translation_workflow_core/contracts/decision_contracts.md`；
- `prompts/templates/D5_D7_quality_gate.prompt.json`；
- `prompts/templates/D8_repair_selection.prompt.json`。

抽取经验：

- 翻译、布局、视觉裁决和修复选择是不同问题，不塞进一个 Prompt；
- D7 只判断源候选差异和病因；
- D8 每轮只选择一个主病因和一个最小修复原子；
- 模型输出必须包含证据、目标区域、置信度和下一步；
- 模型不能绕过病因/修复原子注册表自由发明工具。

P4 将其简化成中文、单病因模板：

- `prompts/page_rhythm_rule_adjudication.zh-CN.json`；
- `prompts/visual_repair_rule_adjudication.zh-CN.json`；
- `prompts/image_graphic_protection_visual_gate.zh-CN.json`。

### 5.4 工具分类和证据边界

来源：

- `pdf_translation_workflow_core/contracts/tool_contracts.md`；
- `pdf_translation_workflow_core/tools/README.md`。

抽取经验：

- 工具分为 probe、planner、renderer、validator、repair 和 orchestrator；
- 每次调用要能说明输入、输出、失败信号和 fallback；
- Poppler 不可用时可退回 PyMuPDF，OCR 不可用时不能假装提取成功；
- PDF 生成后必须渲染为 PNG，结构数据不能替代视觉复核；
- 官方双语参考只用于离线评价，不能成为运行时坐标或文本真值。

落地位置：

- `src/shared_pdf_kernel/`；
- `toolboxes/body/flow_text/single/tools/` 下按工程类别拆分的目录；
- 每页 `render_evidence.json` 与 `previews/`。

### 5.5 反过拟合合同

来源：

- `pdf_translation_workflow_core/contracts/tool_contracts.md` 的 Tool Anti-Overfit Rule；
- `tools/validators/scan_core_overfit.py`。

抽取经验：工具可读取当前页样本事实，但行为不能按文件名、页码、固定文本和固定坐标分支。P4 进一步允许 run-local RepairPatch 保存本页坐标，但不允许这些坐标回写成源码条件。

## 6. 从 `pdf_translation_workflow_lab` 各轮抽取了什么

### 6.1 Round22：隔离运行包和源候选证据

来源：`round22_table_layout`。

保留的经验：

- 输入、输出、报告、预览和离线参考严格分区；
- 每一步保留 source structure、role plan、layout plan、generation evidence、quality gates、decision log 和 operation log；
- 先生成候选并诚实记录产品 FAIL，不能因为过程 PASS 就宣称完成；
- 人工参考页不能被运行时工具读取；
- 结构拆分、障碍边界和源相对字号比固定点值更可迁移。

没有照搬：Round22 的表格/卡片/具体页面启发式和大而全 layout 工具。P4 只复用了运行包、源候选比较和“工具必须由证据驱动”的范式。

### 6.2 Round23：状态优先，但只选修复不算闭环

来源：`round23_state_machine_flow`。

保留的经验：工具调用必须挂在确定状态上，状态退出需要 Artifact 和 Gate。

关键教训：Round23 能选择 `vertical_flow_relayout`，但没有通用 RepairPatch 执行器，所以必须诚实停在 `S_FAIL_QUALITY`。P4 因此要求修复选择、补丁应用、重渲染和复判缺一不可。

### 6.3 Round24：静态分发、RepairPatch 和回滚

来源：`round24_state_machine_repair_patch`。

保留的链路：

```text
质量信号归一
→ Triage 只输出 failure_class
→ 静态 Dispatch 绑定 repair family/tool
→ RepairPatch 填当前页参数
→ 应用补丁
→ 再生成、再裁决
→ 接受或回滚
```

关键教训：Round24 的 `vertical_flow_relayout` 使阻塞数从 80 增至 102，因此被 `REJECTED_ROLLBACK`。这直接形成 P4 的接受规则：目标病因改善还不够，其他硬门禁不能退化。

### 6.4 Round25：因果优先于数量，局部修复优先

来源：`round25_aia_first20_layered_validation`。

保留的经验：

- Triage 按根因优先，不只按 Finding 数量；
- 先修 `text_fit_overflow`，其他病因放入 deferred failures；
- 修复目标从 9 降到 0，但 `cross_slot_overlap` 从 70 增到 78，仍必须回滚；
- 一次修复范围过大时，即使总数略降也可能视觉更差。

P4 因此采用“一页、一个局部转场、一个 RepairPatch”的小步方式。

### 6.5 Round26：自引擎的七类决策证据和能力边界

来源：`round26_contract_driven_selfengine`。

保留的概念：

- evidence basket；
- quality signal ledger；
- problem domain buckets；
- triage result；
- dispatch result；
- repair patch；
- repair acceptance；
- state trace 与 decision graph validation。

关键教训：目标溢出可大幅改善，但碰撞从 140 增到 268，说明缺的是 obstacle-aware 局部工具，不是再写一个更长 Prompt。P4 没有复制七套大文件，而是把必要证据压缩进页级 `rule_decision → repair_patch → application → quality_decision` 链路。

### 6.6 Round27：修复记忆和避免重复无效动作

来源：`round27_obstacle_aware_memory`。

保留的经验：记录已尝试的病因、修复原子、目标范围和 before/after；出现硬回归的组合不得在下一轮原样重复。P4 用有预算的 `repair_trace`、确定性循环上限和“修后立即重算”实现同类约束，没有引入跨文档永久记忆。

### 6.7 Round28：横向稳定、纵向弹性

来源：`round28_column_flow_elastic`。

这是 P4 几何原则的直接来源：

- 按页类型决定排版策略；
- 正文栏宽稳定；
- 普通文本主要纵向增长或收缩；
- 表格、图表、图片和背景保护；
- 只把工程规则无法确定的残余美观问题留给 LLM/人工。

Round28 仍出现目标溢出改善、跨槽碰撞恶化，因此 P4 没有复制其全页弹性流，而是缩小为单列、局部、受预算的纵向修复。

### 6.8 legacy 经验到 P4 代码的对应表

| legacy 经验 | P4 落地 |
|---|---|
| 状态与 Artifact 驱动 | `tools/p4_engine.py` |
| 标准 failure class | `contracts/failure_dispatch_table.json` |
| Triage 与工具分离 | `prompts/*adjudication*` + 静态 dispatch |
| RepairPatch 后再执行 | `tools/orchestrator/repair_loop.py` |
| 当前页信号与局部参数 | `tools/probes/`、`tools/validators/` |
| 目标改善且无硬回归 | `tools/p4_judge.py` 与重渲染复判 |
| obstacle-aware/局部修复 | `section_spacing_reflow.py`、`inline_graphic_control_relayout.py` |
| 源相对布局与纵向弹性 | `tools/p4_layout_planner.py` |
| 过程/产品双结论 | `quality_decision.json`、`run_result.json` |
| 修复记忆和预算 | `repair_trace.json`、deterministic repair trace |

## 7. P4 中 Codex 实际怎样发现并修好问题

### 7.1 `S2P0463`：章节空白过大

观察：候选页第 6 项前出现大块空白，文字本身没有越界，机械 fit 仍可能通过。

Codex 的处理：

1. 对比原文和候选，确认不是正文缺失，而是两个语义区段之间的转场被放大；
2. 从布局计划采集所有相邻转场，不能写死“第 6 项”或该页坐标；
3. 用当前页 IQR 分布判断孤立异常值；
4. 规则识别候选间距 43.8273pt，而源间距和目标间距为 6.416pt；
5. 静态分发到 `section_spacing_reflow`；
6. RepairPatch 将后续容器整体上移 37.4113pt，所有 x 坐标和宽度保持不变；
7. 重跑 `31-p4-s2p0463-regression-20260712T200000Z`，证明规则已进入工程入口。

对应代码：

- `tools/probes/paragraph_spacing_probe.py`；
- `tools/validators/section_spacing_outlier_rule.py`；
- `tools/repairs/section_spacing_reflow.py`；
- `tools/orchestrator/repair_loop.py`。

### 7.2 `S2P0463`：选框随文字重排后漂移

观察：正文节奏修好后，空选框/勾选框仍停在旧位置，甚至挤到下一标题。这个问题不是文字 bbox 修复能解决的。

Codex 的处理：

1. 对比源候选裁剪，确认图形是与文本标签绑定的行内控件，不是背景图片；
2. 从当前页图形尺寸、笔画、位置和绑定容器位移中生成 probe；
3. 工程规则能确定错位时直接产生 `icon_label_misalignment`；
4. 静态分发到 `image_overlay_text_relayout`；
5. 擦除旧控件、按当前容器位移重绘控件；
6. 重新渲染，检查旧位置消失、目标位置完整、背景图片未变。

对应代码：

- `tools/probes/inline_graphic_control_probe.py`；
- `tools/validators/inline_graphic_control_alignment_rule.py`；
- `tools/repairs/inline_graphic_control_relayout.py`；
- `prompts/image_graphic_protection_visual_gate.zh-CN.json`。

### 7.3 `S2P0786`、`S2P0307`：同语义段落左侧起点不一致

观察：译文虽然都在页面内，但属于同一语义层级的段落左侧起点不同，视觉上像多个不相关块。

Codex 的判断：这是水平锚点和语义分组问题，不是通过缩字号或拉宽 bbox 解决。由此固化：普通正文 `x0` 和宽度保持源容器不变，只允许 `y0/y1`、行距、段距和字号在可读下限内纵向适配。

这一经验进入 `p4_layout_planner.py` 与 `p4_judge.py` 的水平硬门禁，也成为用户后来要求“横向基本不动、纵向可以微调”的正式协议。

### 7.4 `S2P0523`：千问返回重复译文

观察：两个不同源容器得到相同译文。不能武断地认为后出现的译文一定错误。

Codex 的处理：

1. 原始模型返回先保存到 `translation_bundle.raw.json`；
2. 规范化层检测可疑重复，记录冲突的两个容器；
3. 对两个冲突容器分别发起单单元重译请求，而不是只重译后一个；
4. 重新组装整页 TranslationBundle 并再次校验 ID、数量和顺序；
5. 将两次请求 ID、延迟、响应哈希和触发原因写入 `translation_retry_trace.json`。

对应实现：`tools/p4_engine.py::_canonicalize_with_targeted_retry`。对应证据：Round32 的 `cases/S2P0523/reports/translation_retry_trace.json`。

### 7.5 Codex 发现“机械 PASS 但视觉 FAIL”时怎么办

如果 PDF 可打开、文字未越界、图片没变，但并排图明显挤压、空洞、层级错误或段落不自然，问题拥有者是 Judge/视觉规则缺失。此时不能把机械 PASS 当产品 PASS，也不能马上扩大到全量；应先：

1. 将视觉现象归到具体 problem domain；
2. 写正例、反例和相邻病因区别；
3. 能工程确定则增加规则；
4. 不能确定则增加单病因千问 Prompt；
5. 将新 Finding 接入分发表和回归测试；
6. 重跑同一页。

## 8. 调试节奏：一次先修好一页

### 8.1 为什么先调一页

一个工具箱刚开始时，不要同时跑四五页并边跑边改。这样会把不同页面的病因混在一起，无法判断是模板、翻译、排版、渲染还是裁决出了问题。

固定节奏如下：

```text
选一个代表页
→ 保存原文、译文、候选和并排图
→ 一次只诊断一个病因
→ 只调用一个修复原子
→ 重渲染并复判当前页
→ 当前页闭环后，将规则和工具写入工程入口
→ 重跑当前页证明无需人工临场干预
→ 再换下一种差异明显的页面
```

### 8.2 何时才扩大样本

满足以下条件后再从 1 页扩到 3 页、10 页和全量：

1. 当前页从统一引擎入口可重复运行；
2. 修复条件不含样本 ID、文件名、页码、公司名、固定文句和固定 bbox；
3. 修复后目标 Finding 消失，其他硬门禁不退化；
4. 同一历史问题重跑时不会再次出现；
5. 输入、输出、Prompt、RepairPatch 和裁决证据都已落盘。

更换样本族或大幅扩大样本时再创建新的隔离 run。对同一批样本增加稳定规则后，可以重跑原 run 的输入集合，不必为每次小修补创建全新项目。

## 9. 几何原则：横向稳定，纵向适配

### 9.1 默认不可横向改动

普通正文默认保持：

- 左边界 `x0` 不变；
- 右边界和 bbox 宽度基本不变；
- 页面宽度、栏道宽度和页边距不变；
- 不通过横向拉宽解决换行；
- 不侵入侧栏、图片、页码、页眉页脚或其他固定结构。

目标语在原有宽度内自然换行。不能因为一行放不下就扩大正文框，也不能让不同语义段落出现不同左边界。

### 9.2 允许纵向调整

主要调整维度是：

- bbox 的 `y0/y1`；
- 段内行距；
- 段间距；
- 同角色文字的统一字号；
- 后续段落沿同一阅读流向上收拢或向下顺延。

英文转中文后文字变短，可以向上收拢过大的段后空白；中文转英文后文字变长，优先向下扩展并推动后续段落。纵向不要求与源页逐像素完全对齐，但不得越出页面、压住下一固定结构或形成明显挤压。

### 9.3 横向例外

只有标题、独立短行或明显未占满的一行，并且当前页几何证明右侧为空白时，才允许小范围横向例外。例外必须记录源 bbox、目标 bbox、栏道右边界、空白证据和角色；普通正文不得使用。

## 10. 先保证语义分块，再谈排版参数

许多“行距不好看”实际是语义分块错误：

- 本应属于同一语义段落的句子被拆成多个容器；
- 不同语义段落被合并成一块；
- 列表编号、复选框或项目符号与正文失去绑定；
- 粗体问题句与普通答案被错误合并；
- 同一语义层级的段落左边界不一致。

正确顺序是：

```text
确认阅读顺序
→ 确认语义段落边界
→ 确认标题、正文、列表、脚注和行内控件角色
→ 确认同角色左边界和字体层级
→ 最后调整字号、行距和段间距
```

如果分块错误，单纯缩字号或压行距只会掩盖问题，并造成后续页面更难维护。

## 11. 病因研判采用规则检查范式

这里借鉴 Drools/专家系统的范式，不引入 Drools 本身：每条病因有明确事实、触发条件、优先级、Finding、修复原子和复判规则。

### 11.1 规则可以直接确定的情况

有稳定、可计算、准确率足够高的直接证据时，由工程规则直接裁决和派发，例如：

- PDF 文字框越过页面边界；
- 普通正文 `x0` 或栏道宽度发生非法变化；
- 两个目标文字框发生明确重叠；
- 阅读顺序反转；
- 图片或背景哈希变化；
- 某个孤立段间距是当前页转场分布的显著异常值；
- 行内控件与绑定文字容器产生可计算的位置偏移。

规则输出统一的 `failure_class`，再由静态分发表映射到一个修复原子和一个工具。规则不得直接散落调用任意工具。

### 11.2 工程证据不能确定的情况

以下问题通常需要千问结合源页与候选页视觉判断：

- 页面是否显得挤压或过度稀疏；
- 同一语义段落的断行是否自然；
- 粗体、标题和正文的层级是否保持；
- 段落节奏是否与源页相近；
- 某个空白是合理留白还是异常断层；
- 候选页是否出现视觉失衡，但工程指标尚未越过硬阈值。

千问不直接执行代码。千问输出标准化病因、目标区域、视觉证据、置信度和候选修复原子；引擎校验输出后，仍通过静态分发表选择已注册工具。若病因或修复原子不在注册表中，必须返回“需要证据”或“能力不足”，不能临时发明工具名。

## 12. 千问应该研判什么

### 12.1 一次只问一个维度或一组相近病因

不要一次让模型同时判断首页、分栏、字体、密度、重叠、图片和修复方案。P4 有效的拆分方式是：

1. 语义分段与阅读顺序；
2. 左侧对齐与字体层级；
3. 段内密度、行距和挤压；
4. 段间距与大面积空白；
5. 行内控件和标签对齐；
6. 最小修复原子选择。

同一请求只包含相近病因，模型更容易给出稳定结论，也方便把结论固化为规则。

### 12.2 必须同时给源页和候选页

纯看候选页只能得到“好不好看”，无法判断是否保持源布局。请求中至少包含：

- 源页截图；
- 候选页截图；
- 当前目标区域的结构化几何事实；
- 本次只允许判断的病因列表；
- 不可动约束；
- 允许选择的修复原子列表。

必要时增加目标区域裁剪图，但不能用记忆中的样例图替代当前页证据。

### 12.3 提示词必须具象化

每个裁决 Prompt 应包含：

- 病因定义；
- 正例：什么视觉和几何证据表示命中；
- 反例：什么情况看似相似但不应命中；
- 与相邻病因的区别；
- 不可动约束；
- 允许的修复原子；
- 严格 JSON 输出结构。

例如判断“段落密度不匹配”时，正例应明确为候选段内显著拥挤、段后却留下大块空白；反例应明确为源页本身就有章节留白、页末自然留白或标题前的合理间隔。

### 12.4 输出必须可执行

建议统一输出：

```json
{
  "verdict": "PASS | FAIL | NEEDS_EVIDENCE",
  "failure_class": "已注册病因或 null",
  "target_regions": ["container_id"],
  "evidence": [
    {
      "dimension": "判断维度",
      "source_observation": "源页观察",
      "candidate_observation": "候选页观察"
    }
  ],
  "suggested_repair_atom": "已注册原子或 null",
  "confidence": 0.0,
  "hard_constraints_respected": true
}
```

不要只返回“页面不好看”“需要调整间距”这类无法派发的自然语言结论。

## 13. 固定的裁决与修复闭环

```text
采集当前页事实
→ 硬规则逐条检查
→ 有直接证据：生成 Finding
→ 无法确定：调用一个单病因千问 Prompt
→ 将结果规范化为 failure_class
→ 静态分发表选择 repair_atom 和 bound_tool
→ 生成 RepairPatch
→ 一次只执行一个修复原子
→ 重新渲染
→ 重采集、重判目标 Finding 和所有硬门禁
→ 改善则接受，否则回滚或进入下一预算轮
```

重要约束：

- 不批量修多个无关病因；
- 不接受只生成了补丁、但没有重渲染和复判的“成功”；
- 目标 Finding 改善但其他硬门禁退化时，补丁仍然拒绝；
- 同一 `(failure_class, repair_atom, target_scope)` 无改善时不得无限重复；
- 修复预算耗尽必须明确产品 FAIL 或能力不足。

## 14. 工具如何固化，避免过拟合

工具代码只能使用当前页事实及其比例、分布和相对关系。禁止写入：

- 样本 ID、文件名和页码；
- 公司名、固定标题、固定句子；
- 针对某页的绝对 bbox；
- 只有某一家文档才能命中的固定点值；
- 从历史候选页记住的视觉结论。

运行时 RepairPatch 可以保存本页容器 ID、坐标、字号和目标间距，因为它们是本次证据；这些值不能反向成为代码分支。

一旦人工修好一页，必须把有效逻辑写回统一引擎入口、规则、分发表和测试。只改某个 run 的输出不算能力沉淀，下一次回归仍会复发。

## 15. 必须保留的运行证据

每个页面至少保留：

- `input/source.pdf`；
- 实际使用的 Prompt；
- 原文事实和页面模板；
- 原始千问返回与规范化译文；
- 初始布局计划；
- 每轮 Finding、RepairPatch 和修复结果；
- `output/candidate.pdf`；
- 源页、候选页和并排预览；
- 过程裁决与产品裁决；
- 最终 `run_result.json`。

过程成功与产品质量必须分开。生成了 PDF 不等于翻译排版通过；千问调用成功也不等于候选页视觉通过。

## 16. P5 多列正文可复用清单

### 16.1 可直接复用

- 页级运行包结构；
- Qwen 配置、页级翻译接口、ID/数量/顺序校验和针对性重试；
- `PageFacts → PageTemplate → TranslationBundle → LayoutPlan → Candidate → Judge` 主链路；
- 原文快照、PDF 可重开、图片/背景保护和字体嵌入；
- Finding、RepairPatch、修复预算和重新裁决合同；
- 静态 failure dispatch table 的结构；
- 单病因中文 Prompt 范式；
- 源候选并排证据与过程/产品双裁决；
- 禁止样本特例的扫描规则。

### 16.2 只能借鉴、不能直接复制

- 单列栏道推导；
- 全页唯一纵向流；
- 单列段间距异常检测；
- 全页统一向下推动后续容器；
- 单列页上的行内控件绑定和移动假设。

### 16.3 P5 必须新增

- `ColumnBand` 和 `column_id`；
- 每个文字容器唯一栏归属及证据；
- 列内阅读顺序和栏间阅读顺序；
- 固定列起点、列宽、栏间距和禁止跨栏边界；
- 每栏独立的纵向流、修复预算和回滚；
- 串栏、跨栏覆盖、错序、列尾异常和修改一栏破坏其他栏的 Finding；
- 两列、三列以及不同密度的正反例与回归集。

P5 的第一条原则仍然是：先选一页，先把栏归属和阅读顺序做对，再翻译、回填和调间距；不要一开始全量跑 23 页。

### 16.4 P5 页首语义组的字形间距与右锚定标题

多列页的页首标题、致送对象、公司性质说明等虽然位于栏正文之前，仍是有顺序的语义组。译文与原文对比时，不能只比较 textbox 的 `y0/y1`：不同字体在文本框内的上、下内缩不同，即使框坐标相同，实际可见行距也可能明显放大或缩小。

因此 P5 固化以下闭环：

1. 从源 PDF 和候选 PDF 提取相邻语义对象的实际字形边缘；
2. 用各自当前字号或行步长归一化，比较原文和候选的相对间距；
3. 超出相对容差时，一轮只选择一个最明显的间距病因；
4. 只纵向移动后一个对象及其同栏、同结构带的下游对象，不改横向 bbox、栏归属和阅读顺序；
5. 从不可变原页重新渲染，再重新提取字形并复判；
6. 任何对象穿越横线、边框等结构锚点时，补丁直接拒绝。

页级右锚定短标题是横向例外：如果当前页证明左侧为空白，可向左扩展 textbox 以避免无意义换行，但渲染必须保持右对齐和原 `x1` 视觉锚点。向左扩框后仍按左对齐绘制，会把标题错误地搬到页面左侧，不能接受。

规则和修复只能使用当前页字形、字号、相对间距、owner 和结构带证据；不得写入样本 ID、标题文字、公司名或固定坐标。

### 16.5 类 Drools 的逐条规则、适用域和静态分发范式

Single 的有效经验不是“让千问一次看完整页并自由选择工具”，而是把视觉修补组织成类 Drools 的规则循环。P5 `paired_row_columns` 穿刺进一步证明：病因匹配正确但适用域错误，同样会把已经对齐的内容修坏。因此每条规则除病因条件外，必须声明页面与区段适用域。

推荐事实键：

```text
page_role
toolbox_key
flow_pattern
flow_band_id
band_mode
band_variant
owner
source/candidate geometry evidence
rendered glyph evidence
repair memory
```

执行循环：

```text
采集当前事实
→ 按优先级从第一条规则开始匹配
→ 同时校验病因条件和适用域
→ 规则可 100% 确定时直接形成 failure_class
→ 工程证据不能确定时，千问只裁决当前一个病因维度
→ 静态 failure_dispatch_table 选择唯一 repair_atom/bound_tool
→ 生成并应用一个 RepairPatch
→ 从不可变源页重渲染并重采集事实
→ 接受或回滚
→ 从第一条规则重新遍历
```

分发表至少包含：

| 字段 | 说明 |
|---|---|
| `failure_class` | 标准化病因，不使用“看起来不好” |
| `applicable_patterns` | 可进入的页级区段范式 |
| `applicable_variants` | 可进入的子范式，例如独立栏或成对行 |
| `region/owner constraint` | 允许修改的区段和所有者 |
| `repair_atom` | 一次只绑定一个最小修复原子 |
| `bound_tool` | 工具箱内的确定实现 |
| `acceptance gates` | 修后必须重新验证的越界、重叠、锚点和视觉信号 |

千问的职责是按规则逐条回答，例如：

- 当前字号是否明显偏小；
- 当前行距是否明显偏紧；
- 当前段距相对原文是否异常；
- 当前短标题是否存在可避免换行；
- 当前区段是否属于成对行而不是独立栏。

千问不得在一个请求中同时判断所有病因，不直接输出坐标，也不自由发明工具。工程侧先生成满足硬约束的候选档位，千问只返回 `too_small / too_tight / acceptable` 等受限结论，分发表再决定是否调用字号回升、行高回升或纵向重排工具。

字号与行距的档位选择不能以“刚好装下”为目标。工程侧应先排除越界、越栏、重叠、结构锚点穿越和不可读档位，再从剩余档位中优先选择更接近原文视觉层级的较大字号。若当前正文区较大、页面仍有充分纵向空白，且放大后不会触发上述硬失败，可以回升字号或行高；但“填满空白”本身不是目标，标题与正文层级、语义段距和原版式留白仍须保持。千问应把“字号是否偏小”和“行距是否偏紧”分开裁决，不能用一个笼统的“不好看”结论同时驱动多个修复原子。

`owner` 只表示栏或通栏归属，不等于局部流作用域。同一页的页首通栏和页尾通栏虽然都可能是 `owner=span`，仍属于不同 `flow_band_id`。任何段距、字号或纵向顺延修复只能影响当前 FlowBand；修后必须重新检查相邻 FlowBand 的文字 bbox 和实际字形均无覆盖，防止页首修复把页尾脚注或结论带入中央多栏区。

成对行的反例已经验证：独立栏的 `post_heading_width_vertical_reflow` 和相对段距修复若不检查 `band_variant`，会单独移动一侧，破坏左右同步甚至制造重叠。正确分发是：

- `independent_columns` 可使用独立栏纵向流和相对段距修复；
- `paired_row_columns` 只能使用左右同步的字号、行高和纵向修复；
- 同一 `owner` 下的不同 FlowBand 不得联动移动；
- 实际字形重叠在所有范式下始终是硬失败，不能因适用域而放行。

### 16.6 实盘核查：怎样把类 Drools 规则交给千问逐条排查

这里的“类 Drools”不是把整套页面和所有工具交给千问，而是先由工程侧形成一条窄规则，再让千问只补足该规则中无法机械确定的视觉条件。推荐的 Rule IR 至少包含：

```text
rule_id
salience
applicable_toolbox
applicable_flow_pattern / band_variant
required_facts
mechanical_when
llm_question（可为空）
allowed_verdicts
failure_class
repair_atom
bound_tool
acceptance_gates
```

执行时必须区分三种情况：

1. `mechanical_when` 已由当前页事实百分百证明：直接形成 `failure_class`，不请求千问；
2. 机械事实只能证明适用域和安全边界，不能证明是否美观：把源图、候选图、已知事实和这一条 `llm_question` 交给千问；
3. 事实不足：输出 `NEED_EVIDENCE`，不得让千问猜坐标、猜工具或绕过分发表。

千问 Prompt 应按规则卡片组织，而不是用一句“请看看哪里不好看”：

```text
当前只检查：正文视觉密度
适用域：body.flow_text.multi / 当前 FlowBand / independent_columns
已由工程确认：横向 bbox、栏归属、图片和结构锚点不可改
逐条问题：
1. 候选字号是否明显低于原文正文视觉层级？
2. 候选多行文字的行距是否明显偏紧？
3. 当前受保护区域之外是否真的存在可用纵向空间？
允许答案：acceptable / too_small / too_tight / too_small_and_tight
禁止：输出坐标、字号、工具名或其他病因
```

即使一次请求检查的是“字号与行距”这一组相近病因，修复阶段仍只能选一个主病因。`too_small_and_tight` 是观察结果，不等于一次同时执行两个工具。引擎必须结合安全候选事实确定优先级，例如先执行一个 `line_height_recovery`，重渲染后再从第一条规则重新检查；若仍 `too_small`，下一轮才允许执行 `font_scale_recovery`。

### 16.7 `S2P0624` 实盘：哪些交给规则，哪些交给千问

本页实际暴露了三个不同病因，不能合成一句“排版不好看”：

| 病因 | 当前页直接证据 | 裁决者 | 修复原子 |
|---|---|---|---|
| 签名区英文单词被拆开 | 源语义为单行；译文在原窄框产生词内断行；同高度左侧为空白；右锚点和签名图可锁定 | 工程规则 | 固定覆盖层向安全空白侧最小扩宽 |
| 同一语义段被拆成两个翻译单元 | 前块无终止标点；同 owner、同字号、同颜色；相邻源行间距属于合并前的当前页行节奏 | 工程规则 | `merge_adjacent_same_owner_fragments` |
| 正文字号偏小、行距偏紧 | 不越界只能由工程证明；“是否明显难读、是否低于原文视觉层级”不能只靠点值确定 | 千问单维度裁决 | 由分发表按主病因选择字号回升或行高回升之一 |

这次实盘还证明，源行节奏必须在开始合并前冻结。若每合并一个容器就重新根据剩余容器计算阈值，容器只剩两个时统计样本不足，阈值会错误收紧，最后一个本应合并的语义片段被遗留。正确做法是：从合并前的完整局部流推导 `owner_line_gap_limits`，整个单病因修复循环复用该页事实，直到规则 PASS。

固定覆盖层的横向扩宽也是规则例外，不是放开普通正文 bbox。只有同时满足“单行语义、当前宽度产生可避免的词内断行、相同高度存在已证明空白、原侧锚点可保持、扩宽后不碰锁定图形”时才允许最小扩宽。普通正文仍保持横向 bbox 不变。

### 16.8 当前 P5 的真实实现边界

截至本次实盘，当前工程入口已经完成：

- 模板阶段逐条执行跨栏拆分、页尾归属和语义碎片合并，修后立即重新匹配；
- 布局阶段按安全档位排除越界，并可在同字号下用真实纵向余量逐级恢复行高和段距；
- 渲染后按实际字形逐条检查段距丢失、段距放大和文字重叠，一轮只执行一个纵向修复原子；
- 千问使用中文 Prompt，对源图和候选图只裁决字号/行距密度维度，并输出受限枚举。

当前尚未完成的是“千问 Finding 到自动修复”的最后闭环：现有入口收到 `too_small`、`too_tight` 或 `too_small_and_tight` 后，只会记录质量 Finding 并判产品失败，还没有经静态分发表选出一个修复原子、重渲染、重新请求千问。因此当前不得描述为“千问已经在运行时选择并执行工具”。

下一步闭环必须保持最小：

```text
千问返回受限 verdict
→ 规范化为一个 primary failure_class
→ 静态分发表选择唯一 repair_atom
→ 从工程预生成且通过硬门禁的候选档中选择一档
→ 从不可变原页重渲染
→ 机械门禁复查
→ 只用同一 Prompt 复判该维度
→ 接受或回滚
```

`S2P0869` 的“大面积留白、字号与行距不协调”属于同一个美化规则族，可以用来回归上述闭环；分割线位置、表格边框和其他结构锚点不属于美化规则，应优先用机械几何证据裁决，不能交给字号/行距 Prompt 一并判断。

### 16.9 固定栏宽内的自然文字流：不能把源 PDF 视觉行当成目标段落

多列正文的栏起点、栏终点和栏宽是版式硬约束；单栏区的左右边界同样是版式硬约束。这里锁定的是**语义文本区域的边界**，不是源语言字形恰好占用的窄 bbox。正文换行应是连续文字流在所属文本区域、目标字体和当前字号下的自然结果，不能为了复刻源 PDF 的行数、字形占宽或抽取块数量而主动制造短行。这个原则对中译英和英译中完全相同。

PDF 抽取器经常把一个自然段拆成“一行一个容器”。如果这些视觉行直接成为翻译单元和布局单元，语言长度变化后就会出现明显错误：上一行仍有大量可用宽度，后续文字却被强制放到下一行；中文尤其容易因译文变短而形成连续短行。此时问题不在栏宽，也不应通过扩栏、缩字号或压行距掩盖，而应先恢复自然段文字流。

横向边界必须先按 owner 映射，再测量目标文字：

| owner / FlowBand | 目标文字可用横向边界 | 禁止行为 |
|---|---|---|
| `span` / 单栏带 | 当前单栏文本区域的完整左右边界 | 把源字形窄 bbox 或源视觉行尾当成硬右边界 |
| `column-N` / 双栏或多栏带 | 该文本块所属栏的完整左右边界 | 跨入相邻栏，或为了复刻源断行而保留栏内短行 |
| `fixed`、`margin` | 经锁定对象和同行障碍证明的局部安全边界 | 套用普通流文本扩宽，破坏覆盖层或页边锚点 |

正确合同是：

1. 单栏文本块可以使用完整单栏文本区域；多栏文本块只能使用所属栏的完整栏宽。两者都不得越过对应区域边界；
2. 源 PDF 中单个文字块或一行字形的 `x1` 不是目标排版边界。只要扩展后仍在上述区域内、同行没有真实文字或图形障碍、且实际测量能减少无必要换行，就应使用完整安全宽度；
3. 模板阶段先在同一栏内识别同一自然段的相邻视觉行，将其合并为一个连续语义单元；
4. 翻译请求以恢复后的自然段为单位，不能把源 PDF 的每条视觉行当成独立句段；
5. 排版阶段使用目标语言真实文字、目标字体和 owner 对应的完整文本区域重新计算换行；
6. 标题、正文、列表项和脚注都服从同一自然换行规则；只有真实语义边界和结构带边界需要保留，抽取器制造的行边界不保留；
7. 自然段最后一行可以合理偏短，但同一自然段仍有后续文字时，前一行不应在存在充足可用宽度的情况下提前结束；
8. 修后重新检查越栏、越界、重叠、阅读顺序、段落边界和实际字形，不能只看容器 fit。

图像障碍也必须区分“underlay”和“右侧障碍”。整页背景、拼贴背景，以及已经承托当前文字框主体的局部横幅/底图，不应阻止文字在同一安全文本区域内横向延伸；真正从当前文字右侧开始、没有承托原文字框的独立图片或图形才是扩宽障碍。判断依据只能来自当前页重叠比例、同行关系和区域几何，不能来自图片内容、标题文本或样本编号。

视觉行合并只能使用可泛化的 multi 事实，例如：同一 `column_id`、相同语义角色、相近字号/字形样式、稳定左锚点、相邻行的相对纵向间距和同一 FlowBand。遇到标题、列表边界、明显段间距、栏切换或结构锚点必须停止。中英文在连接字符上可按语言书写规则处理，但不得按公司名、固定标题、正文词句、数字、页号、样本 ID 或样本专属坐标分支。

建议增加两个独立门禁：

- `source_visual_line_fragmentation`：同一自然段被错误拆成多个翻译/布局单元；
- `premature_line_break`：目标自然段尚有后续文字，当前行却在固定栏宽内留下显著可用空间。

这两个门禁的修复顺序必须是“恢复自然段 → 重新翻译或重组译文 → 固定栏宽重排 → 重渲染复判”，而不是“放宽栏宽 → 强行填满每一行”。该规则还能减少逐视觉行规划造成的容器数量膨胀和重复换行计算，是质量修复，也是多列规划器的通用性能收敛手段。

### 16.10 单栏/多栏混排的纵向边界：按 FlowBand 组合，不按源锚点硬切

分类器把单栏与多栏共存页路由到 `body.flow_text.multi` 时，工具箱必须把页面解释为有序 FlowBand 组合，例如 `single → multi → single`。单栏段落按完整单栏文本区域排版，多栏段落按各自 owner 栏排版；前一带因翻译增高后，后一带可以整体下移。此时多栏带的有效底边是下一内容带的当前输出顶边，而不是它在源页上的旧 `content_bottom`。只要没有侵入下一带、锁定视觉对象或页脚，就不能把这种整体位移误判为越栏。

页尾空间不足时，优先按比例重新分配可变空白，且必须满足以下顺序：

1. 保持每个文字框的横向 owner、文本高度和阅读顺序；
2. 保持同一语义带内已经验证的目标段距下限，优先压缩带间或超出目标值的空白；
3. 以实际页脚译文向上扩展后的顶边作为正文安全底边，而不是固定的页面常数；
4. 每次渲染后用实际字形复查段距和重叠；后续段距修复若下推文本，必须重新执行整页 FlowBand 拟合；
5. 若仅靠空白重分配仍不能容纳，才按通用字号档对整个受影响单栏带等比例收敛，并重新做真实字体测量；不得只缩某个已知句子，也不得低于可读性门禁。

机械 `PASS` 仍不能替代并排图检查。布局 bbox 之间有间隙，不代表字体实际字形一定不重叠；同理，源栏底被越过也不代表发生了视觉冲突。最终结论必须同时满足 owner 边界、FlowBand 顺序、页脚安全区、实际字形段距和人工可见结果。

### 16.11 页眉页脚不是源字形窄框：同一 margin 行必须协同使用安全带

运行页眉和页脚中的自然语言经常与页码、年份、期号、logo 分成多个 PDF span。源语言只有两三个汉字时，抽取得到的 bbox 很窄；若把这个字形占宽直接当成译文硬边界，`Annual Report` 一类英文就会被排成 `Annu / al / Repor / t`。这不是翻译问题，也不是渲染器不会换行，而是 margin 规划器把“源字形 bbox”误当成了“目标语言可用区域”。

页眉页脚的正确合同是：

1. 先按当前页几何识别同一视觉行，再把页码、年份、期号、logo、独立图形等锁定对象投影到该行；锁定对象把这一行切分成若干互不连通的安全横向区段；
2. 落在同一安全区段内的可翻译 margin 片段必须协同规划，保持源阅读顺序、合理间隔和页边基线。不能让每个片段各自扩宽后互相覆盖；
3. 优先保持源字号和原横向锚点。右侧为空时可直接向右扩框；右侧被年份等对象锁住而左侧有空白时，整组只做容纳单行所需的最小安全左移；
4. 锁定对象位于两个可翻译片段之间时，先把两者分到不同安全区段；若其中一个短标签或公司名仍只能断行，而相邻区段存在足够的大空白，则允许把同一行的可翻译片段作为一组迁入最近的可容纳区段。锁定对象仍不得移动，组内阅读顺序必须保持，迁移量取可行方案中的最小值，并在锁定对象与迁移组之间保留至少 `0.35em` 的可见间隔，避免 `2024Clouds` 一类视觉粘连；
5. 原 bbox 只用于定位和建立局部关系，不是目标语言宽度上限。安全边界必须来自当前页的页边、同行文本、图片和图形事实，不能来自样本 ID、公司名、年份文本或样本专属坐标；
6. 空间确实不足时可以在页内向上或向下做自然多行回填，但只能在词边界换行。拉丁单词被拆到多行必须产生 `P5_MARGIN_WORD_FRAGMENTATION / HARD`，不能因为纵向仍放得下就删除失败；
7. 若硬门禁导致 `candidate_pdf = null`，PDF 产物必须缺席；后端异常、provider 错误和诊断文本只能进入 JSON/日志，绝不能绘制到交付 PDF。

这个病因应由工程规则裁决，不需要大模型猜边界：真实字体测量能够证明单行所需宽度，同行障碍能够证明安全区段，bbox 相交能够证明越界或重叠。大模型只应在机械门禁通过之后裁决字号、行距和整体视觉层级。

回归必须同时覆盖四种结构：一是“公司名｜报告名｜锁定年份”，验证同行片段能在年份前协同左移；二是“报告名｜锁定年份｜公司名”的镜像结构，验证优先分区；三是“页码｜长公司名｜锁定年份｜报告名”，验证局部分区放不下时整组可迁入年份后的大空白；四是“公司名｜锁定年份｜报告名｜页码”，验证窄槽中的报告名可与公司名一起迁入年份前的大空白。已知页面只能作为回归证据，运行时代码不得出现这些页面的编号、文字常量或固定坐标。

### 16.12 反过拟合审计不能只扫 Python：prompt 也是运行时代码

“代码里没有样本 ID”只排除了一种最明显的过拟合。分类工具箱至少要审计四层：

1. Python 运行时不得按 case ID、文件名、公司名、已知句子、颜色或绝对坐标分支；布局阈值应由页宽比例、字号、字形宽度和对象相对关系推导；
2. prompt 会直接影响模型行为，也属于运行时代码。正反例必须使用虚构的结构等价文本，不能复制 development/regression 页面的真实标题或句子；
3. 测试可以引用已知样本作为证据，但必须另有镜像结构、不同片段顺序和无安全空间等合成用例，证明规则依赖结构而不是页面身份；
4. 使用同一批输入做全量回归只能证明“这批没有退化”，不能证明泛化。正式的无过拟合结论必须等 workflow 冻结后运行未见 holdout，并同时做机械门禁与源页/结果页视觉比较。
5. 发布前必须在“远端 HEAD + 本次 staged changes”的隔离树中复跑关键测试。脏工作区里的兄弟工具箱改动可能提前消除某个病因、改变 repair 记录或显著改变运行时间；这种混合结果不能作为本次提交自洽的证据。

像 `0.35em` 间隔、页宽百分比 padding、同行中心线容差这类相对阈值可以成为通用合同；某页的 x/y 坐标、年份文本或公司名不能。静态扫描应同时覆盖 `tools/` 与 `prompts/`，并通过测试禁止样本签名重新进入这两个运行时目录。

## 17. 新分类工具箱启动清单

### 17.1 开始编码前

- [ ] 已读该分类计划任务、Gate 和退出条件；
- [ ] 已看分类目录的联系图或代表页，知道正例和反例；
- [ ] 已列出不可动对象和允许调整的维度；
- [ ] 已列出可直接复用、只复用合同、明确不复用的旧能力；
- [ ] 已从 development 选择一个结构清晰的首页；
- [ ] holdout 尚未用于调试。

### 17.2 首页基线

- [ ] 原文快照和哈希已保存；
- [ ] PageFacts、模板、翻译请求、原始译文和布局计划已落盘；
- [ ] 已生成候选 PDF 和源候选并排图；
- [ ] 已分别记录过程结论和产品结论；
- [ ] 即使失败，也保留了完整基线。

### 17.3 Codex 第一次研判

- [ ] 先查翻译，再查模板，再查布局，再查渲染，最后查裁决；
- [ ] 已确定一个主病因和拥有模块；
- [ ] 已判断该病因能否用直接工程证据确定；
- [ ] 如需千问，Prompt 只判断一个维度或一组相近病因；
- [ ] 千问输出可规范化为已注册 failure class；
- [ ] 没有让千问自由发明工具。

### 17.4 修改和复判

- [ ] 只改病因拥有模块；
- [ ] 只执行一个 RepairPatch；
- [ ] 工具没有样本 ID、页码、固定文本、固定坐标特例；
- [ ] 重渲染并重采集了当前页证据；
- [ ] 目标 Finding 改善；
- [ ] 其他硬门禁没有退化；
- [ ] 失败补丁已回滚而不是继续叠加。

### 17.5 进入下一页前

- [ ] 有效逻辑已进入统一引擎入口；
- [ ] 病因、分发表、Prompt 和工具均已落盘；
- [ ] 已增加回归测试；
- [ ] 从正式入口重跑当前页仍通过；
- [ ] 新页面与当前页存在明确结构差异，能验证下一项能力；
- [ ] 尚未因为“机械 PASS”跳过视觉复核。

### 17.6 必须停下来而不是扩样的情况

- 机械 PASS 但并排图明显难看；
- 尚不能判断问题属于翻译、模板、布局、渲染还是裁决；
- 修复需要文件名、页码、固定句子或固定 bbox 才能命中；
- 同一问题仍依赖 Codex 手工修改 run 输出；
- 一个修复改善目标病因却引入新的硬回归；
- 新分类需要修改公共内核，但还没有回放已稳定工具箱；
- holdout 已被用于调参；
- 只生成了 RepairPlan，没有执行、重渲染和复判。
