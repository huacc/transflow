# body.layout_owner 二级排版所有权初判

上一级已经确认 `page.role=body`。你本次只判断哪种排版结构拥有页面的主要翻译排版权，不判断栏数、密度、语种或具体工具参数。

你只能使用匿名页面截图和原始结构证据。禁止根据文件名、路径、人工标签或旧分类结果判断。

图片只用于分类。图片像素及图片内部文字、数字、图例和标签不翻译、不重排；独立 PDF 文字对象仍可处理。

## 当前节点

```text
node_key = body.layout_owner
allowed_choices = flow_text | table | chart | diagram | anchored_blocks | composite
```

允许返回 `INCONCLUSIVE`。`anchored_blocks` 和 `composite` 都不是兜底项。

高优先级边界：会计准则、制度或修订项目清单不是普通正文列表。截图中若左列逐行给出项目名称、右列在同一行给出对应说明或生效信息，这些行必须保持横向配对，归属 table；页面同时存在独立连续正文时选择 `composite`。不得以“清单属于正文的一部分”或“左右形成两条阅读栏”为由改判 `flow_text`。

## 分类定义

### flow_text

一条或多条稳定文字阅读流占主体：

- 段落、列表、日期事件或双语对照能够按稳定顺序阅读；
- 可以是单栏、二栏、三栏或更多栏；
- 小表、小图或照片只是附件，不拥有主要排版权。

三列日期事件仍是 `flow_text`。本节点不判断具体栏数。

大面积固定照片或图片拼贴中只有一个局部连续正文区时仍是 `flow_text`；图片本身不可编辑，不得把多张图片误当成多个可编辑卡片。其具体视觉锚定方式由下一级判断。

带可翻译的行标题、列标题或项目标签，且数字必须保持行列绑定的小表，不是可忽略附件。表格面积小不等于可由正文工具处理；只要翻译这些标签时必须保持行列与数值对应，表格就需要独立 owner，页面另有独立连续正文时选择 `composite`。
即使只有一行数据，只要存在年份或期间列头、计量单位及横向绑定数值，也仍是语义表格；页面另有正文时选择 `composite`。
`TABLE1.count=0` 或 `BTABLE1.aligned_row_count=0` 不能否定截图中清楚可见的小型无边框表格：单行数据可能达不到机械检测器的重复行阈值。截图若同时显示共享年份/期间列头、计量单位、项目行标题和按列绑定的数值，必须保留 table owner。
多行重复的“制度、准则或修订项目—对应说明/生效信息”两列配对也是语义表格，即使没有显式表头、边框或数值列，也不能当成多栏正文；页面另有连续正文时选择 `composite`。

### table

主体信息绑定于行、列、表头和单元格：

- 大部分可翻译文字属于单元格；
- 表格网格或规则行列占主要面积/信息量；
- 少量标题、脚注或说明不改变表格所有权。

表格不要求存在外框线。若文字形成至少四个重复行，并在稳定列锚点上逐行配对，即使没有网格线，也属于无边框表格，不能判断成单栏或多栏正文。

结构证据中的 `BTABLE1` 只表示原生文字存在重复对齐候选。对齐置信度不能单独证明表格语义：
KPI 卡片、公司资料块和多栏列表也可能高置信对齐。只有工程规则同时确认稳定的数据行、数据列、
表头与逐行对应关系并给出直接高置信裁决时，才把它作为不可覆盖的表格直接证据。

公司资料块没有跨区块共享的行表头，也没有同一数据列在多行中的逐行对应；常见形态是两栏内分别
排列“字段标题 + 一项或数项内容”，各区块有自己的标题和高度。这属于 `anchored_blocks`，不能仅
因左右对齐或字段很多判为 `table`。

### chart

主体是柱、线、扇区、坐标轴、图例或数据标签表达的数值关系：

- 柱状图、饼图、折线图等占页面主体；
- 若整个图表位于图片中，仍可判 `chart`，但图片内部内容不会被翻译。

只有大数字或 KPI，没有真实数据图形时不是 `chart`。

多个并列或上下排列的柱图、折线图、环图等面板仍属于 chart；不能因为每个图表有标题、边界或
注释就把面板误当成 anchored_blocks。图表标题、图例、轴/类别标签、数据注释，以及
解释同一组指标的段落仍归 chart owner。只有与图表主题和视觉容器独立、可单独连续阅读的实质正文带，才可能
形成 `flow_text + chart` 的 composite。

在选择 `flow_text + chart` 前，必须能在 `reason_summary` 中指出至少一个与所有图表主题无关、
位于独立正文区域且脱离图表仍可连续阅读的正文带。若文字只是逐项解释收入、成本、比例、趋势等
同页图表指标，无论段落多长，均由 chart 工具作为图表注释/说明处理，选择 `chart`。

段落长度本身不产生 flow owner。位于图表上方或旁边、逐段解释同页图表数据、趋势或指标的长段落，
仍可作为 chart 的 `ANNOTATION` 在固定注释槽中处理。只有移除图表标题、图例、标签和这些关联注释后，
页面仍剩下具有自身标题与阅读顺序、需要独立扩容的叙述正文流，才选择 `composite`。

### diagram

主体是组织结构、股权关系、流程、时间关系或地图标注：

- 节点、连线、层级、方向或空间位置关系占主导；
- 不要求存在数据坐标轴；
- 组织架构图、项目地图不是 `chart`。
- 具有多个实质节点、箭头、连线、层级或固定空间顺序的结构需要 diagram owner；主体节点或箭头
  不能降为正文附件。只有很小的装饰图标才可以忽略。
- 只要独立结构带中存在多个带文字节点和方向箭头/连接关系，该结构就需要 diagram 工具；不能因
  上下另有更多正文就把它称为“视觉附件”。同时存在独立连续正文时选择 `composite`，没有独立
  正文时选择 `diagram`。
- `diagram` 单一 owner 只适用于结构图之外没有实质连续正文，或其他文字只是结构标题、节点说明、
  图注和短引言。若理由同时确认实质正文与主体结构图都不可忽略、都需不同处理，`selected_child`
  必须是 `composite`，不能在理由中承认双 owner 却返回单一 `diagram`。
- 时间线或里程碑节点可以包含完整多句段落。若多个故事节点沿共同竖线、轨道、箭头或固定顺序排列，
  标题和段落属于节点内容，整体仍由 `diagram` 拥有；不能因为节点内部文字很长就把连接关系降为装饰。

### anchored_blocks

多个独立 KPI、指标、卡片或说明块共同承载内容：

- 各块有独立边界和锚点；
- 块之间不能形成一条或多条稳定纵向正文流；
- 没有表格单元格、数据图表或结构图连线所有权。

稳定排列的多栏正文不是 `anchored_blocks`。
如果多个 KPI 区块重复采用“指标标题 + 大数字/百分比 + 本区块说明”，即使区块整齐排成两列或三列、每块内部含有段落，也应判断各块之间能否连续阅读；不能跨块连续阅读时属于 `anchored_blocks`，不是 `flow_text`。
固定照片、图片拼贴或背景图不是 `anchored_blocks` 的可编辑卡片。只有一个局部正文容器覆盖或贴靠大面积视觉时，应判 `flow_text`。

### composite

两种或以上结构都含有必须保持自身关系的可翻译内容，需要不同排版工具修复，且任一结构都不是可忽略附件。是否需要独立 owner 由关系约束决定，不要求两种结构面积或文字量势均力敌。

当前典型组合：

- 正文 + 主体表格；
- 独立 KPI/业务卡片 + 主体数据图表；
- 正文 + 组织/股权结构图；
- 财务表 + 经营图表。

`composite` 只允许进入当前 taxonomy 能表达的五种组合：`flow_text + table`、
`anchored_blocks + chart`、`chart + table`、`flow_text + chart`、`flow_text + diagram`。
`flow_text + anchored_blocks 不是允许的复合组合`；卡片页中的引言、卡片说明和页脚应继续归
anchored_blocks/shared owner，不能为凑复合类别虚构 chart。观察到的两种结构不属于上述组合时，
选择实际主导的单一 owner 或返回 `INCONCLUSIVE`。

主体散点图、矩阵图之外若另有占据显著区域的分类议题清单，每项是可独立翻译的议题块而不只是
短图例标签，则图表与信息块需要不同 owner，应选择 `composite`，后续映射到
`anchored_blocks_chart`。

若小表、小图或说明没有需要翻译的关系化标签，或由主体 owner 处理也不会破坏关系，才可视为附件并选择拥有主体的单一类型。不得仅因表格面积较小就把它降为附件。

## 判断方法

1. 找出承载主要信息和主要可编辑文字的结构。
2. 判断文字允许在正文流、单元格、图表标签、结构节点还是独立卡片内移动。
3. 检查第二种结构是否也需要独立工具处理。
4. 第二种结构即使面积较小，只要翻译时必须由独立 owner 保持行列、图表或节点关系，就与主体共同判 `composite`。
5. 对 chart/composite，先做“独立正文证明”：找不到与图表主题无关的正文带时必须选 `chart`。
6. 对 diagram，带文字节点和箭头形成独立结构带即拥有排版权，不能按面积或文字总量降级为附件。
7. 最后核对结果字段与理由：理由同时确认实质正文与主体结构图时必须选择 `composite`。

## 具象化正例与强反例

### L1 flow_text 正例

页面下方分三栏排列日期和事件，每栏都能从上到下连续阅读，栏间有稳定空白。判断 `flow_text`。

### L2 flow_text 强反例

页面看起来有三列文字，但文字都绑定“风险、影响、缓解措施”表头和规则行。判断 `table`，不是多栏正文。

### L3 table 正例

财务表覆盖页面主体，包含年度列、项目行和大量数字，只有一小段脚注。判断 `table`。

另一正例：整页没有网格线，但十余行内容均重复采用“术语—指—释义”三个固定列锚点。逐行配对和列对齐必须保留，判断 `table`，不是多栏正文。

### L4 chart 正例

页面包含六个柱状图，每个都有年份轴、柱形和数值标签。判断 `chart`。

### L5 chart 强反例

页面只有多个大号金额、百分比和说明卡片，没有柱、线、扇区、坐标轴或图例。判断 `anchored_blocks`。

### L6 diagram 正例

中国地图上有七个位置节点、编号和地点说明。判断 `diagram`。

### L7 anchored_blocks 正例

页面由多个 KPI 大数字、说明块和业务卡片组成，各块独立，没有稳定正文流。判断 `anchored_blocks`。

另一正例：页面形成四宫格，每格都有独立指标标题、分隔线、大号百分比和说明段落。阅读完一格后不能自然续读到相邻格，判断 `anchored_blocks`，不能因为视觉上排成两列就判断 `flow_text`。

### L8 anchored_blocks 强反例

双语左右栏各自连续向下阅读，即使包含很多短段落也应判 `flow_text`。

另一强反例：页面大部分是固定照片拼贴，只有右下角一个连续正文区域。图片不翻译、不重排，因此不能把每张照片当成独立卡片；判断 `flow_text`，后续再判断视觉锚定拓扑。

### L9 composite 正例

上半页是大段财务说明，下半页是占较大面积的多行多列表格，两部分都需要翻译和独立修复。判断 `composite`。

另一正例：页面上下都有连续财务说明，中间只有一项数据，但该项明确绑定两个年度列、各自计量单位和两个数值。即使 `TABLE1`/`BTABLE1` 未检出、表格面积很小，也判断 `composite`，不是 `flow_text`。

无边框表格同样适用：上半页是连续责任说明，下半页有十余行“基金名称—成立日期”稳定对齐数据，两部分都不可忽略，判断 `composite`。

另一正例：页面同时包含多组独立业务卡片和一个占显著区域的环形图，两者均不可忽略且需要不同工具处理。判断 `composite`。

## 输出

只返回一个 JSON 对象：

```json
{
  "node_key": "body.layout_owner",
  "status": "DECIDED",
  "selected_child": "flow_text",
  "confidence": 0.92,
  "evidence_refs": ["IMG1", "TEXT1", "B003"],
  "reason_summary": "页面主体形成两条稳定正文阅读流"
}
```

不得输出具体栏数、密度、工具名或隐藏推理。证据不足时返回 `INCONCLUSIVE` 和 null。
